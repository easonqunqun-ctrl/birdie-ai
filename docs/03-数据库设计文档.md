## 一、设计原则

1. **主键策略**：使用带前缀的 UUID 字符串（如 `usr_`, `ana_`, `chat_`），便于识别和调试
2. **时间字段**：所有时间使用 `TIMESTAMP WITH TIME ZONE`，存储 UTC 时间
3. **软删除**：核心业务表使用 `deleted_at` 字段实现软删除
4. **审计字段**：所有表包含 `created_at`、`updated_at` 字段
5. **JSON 字段**：复杂嵌套数据使用 `JSONB` 类型，并建立 GIN 索引
6. **枚举类型**：使用 PostgreSQL ENUM 类型或 VARCHAR + CHECK 约束

### 3.2 analysis_quotas（分析配额表 — 月度）

```sql
CREATE TABLE analysis_quotas (
    id                      VARCHAR(32) PRIMARY KEY,
    user_id                 VARCHAR(32) NOT NULL REFERENCES users(id),
    
    quota_month             VARCHAR(7) NOT NULL,          -- 格式: 2026-04
    used                    INT DEFAULT 0,                -- 已使用次数
    total                   INT DEFAULT 3,                -- 免费用户3次，会员-1（代表无限）
    bonus                   INT DEFAULT 0,                -- 邀请/分享获得的额外次数
    
    created_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    CONSTRAINT uq_analysis_quota UNIQUE (user_id, quota_month)
);

CREATE INDEX idx_analysis_quotas_lookup ON analysis_quotas(user_id, quota_month);
```

**业务规则**：
- 每次请求分析时查找当月记录，不存在则自动创建
- 剩余次数 = `total + bonus - used`，`total = -1` 代表无限
- 会员过期时，当月已有记录的 total 保持不变（不回退已享受的权益），下个自然月开始恢复为 3
- 分析失败退回配额：`used -= 1`

### 3.3 chat_quotas（对话配额表 — 每日）

```sql
CREATE TABLE chat_quotas (
    id                      VARCHAR(32) PRIMARY KEY,
    user_id                 VARCHAR(32) NOT NULL REFERENCES users(id),
    
    quota_date              DATE NOT NULL,                -- 格式: 2026-04-14
    used                    INT DEFAULT 0,                -- 已使用轮次
    total                   INT DEFAULT 5,                -- 免费用户5次，会员-1（代表无限）
    
    created_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    CONSTRAINT uq_chat_quota UNIQUE (user_id, quota_date)
);

CREATE INDEX idx_chat_quotas_lookup ON chat_quotas(user_id, quota_date);
```

**业务规则**：
- 每次发送对话时查找当日记录，不存在则自动创建
- 剩余次数 = `total - used`，`total = -1` 代表无限
- 每日 00:00（UTC+8）自然重置（通过创建新记录实现，无需定时任务）
- 旧的配额记录按 90 天策略清理

### 3.4 swing_analyses（挥杆分析表）

> **M2 实现偏差说明（2026-04 W3-W4 核查）**  
> 实际 alembic `0001_initial_schema.py` 的落库与下方 SQL 有以下差异，MVP 范围内**以迁移为准**：
>
> 1. **诊断与建议独立建表**（规范化，而非内嵌 JSONB）：
>    - `detected_issues JSONB` → 移到独立表 `analysis_issues`（见 §3.5）
>    - `recommendations JSONB` → 移到独立表 `analysis_recommendations`（drill_id / target_issue / sort_order 三列，详细步骤由前端 `client/src/constants/drillLibrary.ts` 查表）
> 2. **延后字段**：`feature_data JSONB`、`model_version`、`processing_started_at`、`processing_duration_ms`、`deleted_at` 暂未落库 —— 进度曲线/对比/软删除挂 **W7 会员能力** 再补迁移。
> 3. **新增字段**：`is_sample BOOLEAN DEFAULT FALSE`（**示例视频体验**，默认列表查询 `WHERE is_sample = FALSE`）—— 在 `0001_initial_schema.py` 中已建好，无需额外迁移。
> 4. **GIN 索引 `idx_analyses_issues`** 因 `detected_issues` 字段不存在而取消；查询 issues 走 `analysis_issues.analysis_id` 上的普通 B-Tree 索引。
> 5. **`quality_warnings`（`0015_swing_analysis_quality_warnings`）**：`JSONB`，存 AI 引擎返回的非阻断质量提示 machine codes 列表（如 `["low_light","camera_shake"]`），详见 [`docs/02` §3.4](./02-API接口设计文档.md)。

```sql
CREATE TABLE swing_analyses (
    id                  VARCHAR(32) PRIMARY KEY,         -- 格式: ana_<nanoid>
    user_id             VARCHAR(32) NOT NULL REFERENCES users(id),
    
    -- 输入
    video_url           VARCHAR(512) NOT NULL,           -- 原始视频 COS 路径
    video_duration      DECIMAL(5,2),                    -- 视频时长（秒）
    video_file_size     BIGINT,                          -- 文件大小（字节）
    camera_angle        VARCHAR(20) NOT NULL,            -- face_on / down_the_line
    club_type           VARCHAR(20) NOT NULL,            -- driver/iron_7/wedge/...
    
    -- 处理状态
    status              VARCHAR(20) NOT NULL DEFAULT 'pending',
    stage               VARCHAR(30),                     -- 当前处理阶段
    stage_progress      INT DEFAULT 0,                   -- 阶段进度 0-100
    error_code          INT,                             -- 失败错误码
    error_message       VARCHAR(500),                    -- 失败错误消息
    quota_refunded      BOOLEAN DEFAULT FALSE,           -- 是否已退回配额
    priority            VARCHAR(10) DEFAULT 'standard',  -- standard/priority
    
    -- 输出 - 评分
    overall_score       INT,                             -- 综合评分 0-100
    score_change        INT,                             -- 与上次的变化
    phase_scores        JSONB,                           -- 各阶段评分详情
    /*
    phase_scores 结构:
    {
      "setup": {"score": 82, "details": {...}},
      "backswing": {"score": 74, "details": {...}},
      "top": {"score": 80, "details": {...}},
      "downswing": {"score": 65, "details": {...}},
      "impact": {"score": 79, "details": {...}},
      "follow_through": {"score": 85, "details": {...}}
    }
    */
    
    -- 输出 - 诊断
    detected_issues     JSONB DEFAULT '[]',              -- 检测到的问题列表
    /*
    detected_issues 结构:
    [
      {
        "type": "casting",
        "name": "抛杆（Casting）",
        "severity": "high",
        "confidence": 0.92,
        "description": "...",
        "key_frame_url": "...",
        "key_frame_timestamp": 1.8,
        "metrics": {
          "wrist_release_angle": 35,
          "ideal_range": [15, 25]
        }
      }
    ]
    */
    
    -- 输出 - 建议
    recommendations     JSONB DEFAULT '[]',              -- 训练建议列表
    /*
    recommendations 结构:
    [
      {
        "drill_id": "drill_towel_arm",
        "reason": "修复 casting 问题",
        "priority": 1
      }
    ]
    */
    
    -- 输出 - 视觉资源
    skeleton_video_url  VARCHAR(512),                    -- 骨骼叠加视频 URL
    skeleton_data_url   VARCHAR(512),                    -- 骨骼数据 JSON 文件 URL
    share_card_url      VARCHAR(512),                    -- 分享卡片 URL
    thumbnail_url       VARCHAR(512),                    -- 缩略图 URL
    
    -- 输出 - 阶段时间戳
    phase_timestamps    JSONB,
    /*
    phase_timestamps 结构:
    {
      "setup": {"start": 0.0, "end": 0.8},
      "backswing": {"start": 0.8, "end": 1.5},
      ...
    }
    */
    
    -- 输出 - 原始特征数据（用于进步曲线和对比）
    feature_data        JSONB,
    /*
    feature_data 结构:
    {
      "spine_angle_address": 32.5,
      "spine_angle_impact": 30.1,
      "hip_rotation_backswing": 42.0,
      "hip_rotation_downswing": 55.0,
      "shoulder_rotation": 88.5,
      "x_factor": 46.5,
      "wrist_release_angle": 35.0,
      "swing_tempo_ratio": 2.8,
      "head_movement_cm": 3.2,
      ...
    }
    */
    
    -- 处理信息
    processing_started_at   TIMESTAMPTZ,
    processing_duration_ms  INT,                         -- 处理总耗时
    model_version           VARCHAR(20),                 -- 使用的模型版本
    
    analyzed_at         TIMESTAMPTZ,                     -- 分析完成时间
    is_sample           BOOLEAN NOT NULL DEFAULT FALSE,  -- 是否为示例视频体验（不计配额、不入列表，M2-T6）
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ,                     -- 用户侧软删除（可空；Alembic 0010）；列表/曲线筛「未删除」走部分索引 0011
    
    CONSTRAINT chk_camera_angle CHECK (camera_angle IN ('face_on', 'down_the_line')),
    CONSTRAINT chk_status CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
    CONSTRAINT chk_priority CHECK (priority IN ('standard', 'priority')),
    CONSTRAINT chk_overall_score CHECK (overall_score IS NULL OR (overall_score >= 0 AND overall_score <= 100))
);

-- 索引
CREATE INDEX idx_analyses_user_id ON swing_analyses(user_id);
CREATE INDEX idx_analyses_user_date ON swing_analyses(user_id, created_at DESC);
CREATE INDEX idx_analyses_user_status ON swing_analyses(user_id, status);
CREATE INDEX idx_analyses_status ON swing_analyses(status) WHERE status IN ('pending', 'processing');
CREATE INDEX idx_analyses_created_at ON swing_analyses(created_at);
CREATE INDEX idx_analyses_club_type ON swing_analyses(user_id, club_type);
CREATE INDEX idx_analyses_deleted ON swing_analyses(deleted_at) WHERE deleted_at IS NULL;
CREATE INDEX idx_analyses_issues ON swing_analyses USING GIN (detected_issues);
```

### 3.5 chat_sessions（对话会话表）

```sql
CREATE TABLE chat_sessions (
    id                      VARCHAR(32) PRIMARY KEY,     -- 格式: chat_<nanoid>
    user_id                 VARCHAR(32) NOT NULL REFERENCES users(id),
    context_analysis_id     VARCHAR(32) REFERENCES swing_analyses(id),
    
    message_count           INT DEFAULT 0,
    last_message_at         TIMESTAMPTZ,
    last_message_preview    VARCHAR(200),                 -- 最后一条消息预览
    
    -- LLM 上下文管理
    system_prompt_version   VARCHAR(16),                  -- system prompt 版本（M3-T2 迁移 0002 加入）
    
    created_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at              TIMESTAMPTZ
);

CREATE INDEX idx_chat_sessions_user ON chat_sessions(user_id, created_at DESC);
CREATE INDEX idx_chat_sessions_active ON chat_sessions(user_id, last_message_at DESC) WHERE deleted_at IS NULL;
```

**M3 实现偏差说明**：

- `system_prompt_version` 字段由迁移 `alembic/versions/0002_chat_system_prompt_version.py` 添加，**字段长度 `VARCHAR(16)`**（原 DDL 写的 10 太窄，无法容纳 `"v1-with-ctx"` 这类版本号；以迁移为准）
- 字段 **nullable**：T1 时期建的 session 没有这个字段，历史数据迁移后为 `NULL`，代码侧读取时按"未记录"处理，不影响 LLM 调用
- 每次**创建新会话**时后端按当前 `chat_prompt.SYSTEM_PROMPT_VERSION`（目前 `"v1"`）写入；会话生命周期内不再改动（即使 prompt 升级了，旧会话保持写入时的版本，方便后续 A/B 评估）

## 七、W7 实现偏差说明（2026-04）

§3.7-3.14 的表在 W7 真实落地时与设计稿存在以下小幅差异，记录方便后续 ALTER 或设计稿回修：

| 表 | 实现差异 | 原因 |
|---|---|---|
| `training_plans` | 加了 `(user_id, plan_week)` 唯一约束（设计稿未写） | 保证"一周一个计划"的业务不变式；增量分析时 upsert 而非插多条 |

| `invitations` | 实现时 `status` 只有 `registered/valid` 两个值（设计稿一致） | W7 实装的 5-人档奖励靠"valid 数 ≥ 5 且未发过奖"判定，不需要 `rewarded` 状态；`bonus_granted` 字段已够用 |

### 7.1 配额动态重算（W7-T1 新增行为）

`analysis_quotas` / `chat_quotas` 在 W7 之前是"创建时冻结 total"的，W7 在两处 hook 里**动态写 total**：

- **激活会员**（`payment_service.activate_membership` 在 `_mark_paid` 成功后）：
  把**当月** `analysis_quotas.total = -1`（无限）、**当日** `chat_quotas.total = -1`（无限）
- **过期降级**（`payment_service.ensure_membership_valid` 在每次 `get_current_user` 前懒检查）：
  若 `membership_expires_at <= now()` → `users.membership_type='free'` + `auto_renew=false`；同时把当月/当日配额中仍为 `-1` 的 `total` 回置为 `FREE_USER_MONTHLY_ANALYSES=3` / `FREE_USER_DAILY_CHATS=5`（`used` / `bonus` 不动）
- **新月/新日配额**（`quota_service.get_or_create_analysis_quota` / `get_or_create_chat_quota` 在首次消费时）：
  根据当前 `user.membership_type` 决定 `total` 的初始值（会员=`-1`、免费=`3` / `5`）

**关键取舍**：会员期内消费分析时，`quota.total<0` 的分支**不会**累加 `used`（见 `consume_analysis_quota`），所以会员期的消费不留"已用"痕迹；过期回落后，`used=0 + total=3 + bonus=0 → remaining=3`，得到一个干净的本月新额度。

`analysis_remaining(quota)` 的计算：若 `total<0` 直接返回 `9999`（前端识别为"无限"）；否则 `max(0, total + bonus - used)`。
