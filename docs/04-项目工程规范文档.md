### 1.5 客户端视觉规范（与产品白皮书一致）

客户端（`client/` 及后续 RN 壳工程）的视觉实现须与 [`小鸟AI高尔夫-产品设计白皮书.md`](../小鸟AI高尔夫-产品设计白皮书.md) **§7.2 产品视觉规范（权威，全端统一）** 对齐，摘要如下：

- **品牌三色**：**深绿**（主色/定调）、**白**（阅读与卡片基色）、**金**（小面积点缀：会员、高分、关键数据等）。
- **禁止**：以高饱和浅绿、荧光绿或「Tailwind 默认 emerald」等作为主品牌主色单独替代文档中的深绿体系，除非经产品与设计书面变更规范。
- **工程落地**：全局色值建议以 SCSS/CSS 变量或设计 Token 集中定义（如 `app.scss`），页面与组件引用 Token，避免魔法数字散落。
- **双端一致**：微信小程序与 React Native 共用同一套 Token 语义；若某端渲染差异导致色偏，以白皮书对比度与可读性要求为准做适配。

## 三、编码规范

### 3.1 Python 后端规范

**代码风格**：

- 遵循 PEP 8
- 使用 `ruff` 作为 linter 和 formatter
- 行宽限制：120 字符
- 使用 Type Hints（所有函数参数和返回值）

**命名规范**：

| 类型 | 规范 | 示例 |
|------|------|------|
| 模块/文件 | snake_case | `user_service.py` |
| 类 | PascalCase | `UserService` |
| 函数/方法 | snake_case | `get_user_by_id()` |
| 变量 | snake_case | `user_count` |
| 常量 | UPPER_SNAKE_CASE | `MAX_VIDEO_DURATION` |
| Pydantic Model | PascalCase + 后缀 | `UserCreateRequest`, `UserResponse` |
| SQLAlchemy Model | PascalCase (单数) | `User`, `SwingAnalysis` |
| API Router | snake_case (复数) | `/users`, `/analyses` |

**项目配置**（pyproject.toml 关键部分）：

```toml
[tool.ruff]
line-length = 120
target-version = "py311"

[tool.ruff.lint]
select = ["E", "F", "I", "N", "W", "UP", "B", "SIM", "RUF"]

[tool.ruff.lint.isort]
known-first-party = ["app"]

[tool.pytest.ini_options]
asyncio_mode = "auto"
testpaths = ["tests"]
```

**目录分层职责**：

| 层 | 职责 | 示例 |
|----|------|------|
| api/ | 接收请求、参数校验、调用 service、返回响应 | 不含业务逻辑 |
| schemas/ | 请求/响应数据结构定义和校验 | Pydantic 模型 |
| services/ | 核心业务逻辑 | 调用 model、第三方集成、编排流程 |
| models/ | 数据库模型和查询 | SQLAlchemy 模型 |
| core/ | 横切关注点 | 安全、缓存、异常、中间件 |
| tasks/ | 异步后台任务 | 视频分析、通知发送 |
| integrations/ | 第三方服务封装 | 微信、COS、LLM |

### 3.2 TypeScript 前端规范

**代码风格**：

- 使用 ESLint + Prettier
- 使用函数式组件 + React Hooks
- 行宽限制：100 字符

**命名规范**：

| 类型 | 规范 | 示例 |
|------|------|------|
| 组件文件 | PascalCase | `ScoreCard.tsx` |
| 工具文件 | camelCase | `formatDate.ts` |
| 组件名 | PascalCase | `ScoreCard` |
| Hook | camelCase + use 前缀 | `useAuth` |
| 类型/接口 | PascalCase + I 前缀（接口）或无前缀 | `IUserInfo` 或 `UserInfo` |
| 常量 | UPPER_SNAKE_CASE | `API_BASE_URL` |
| CSS 类 | kebab-case (BEM) | `score-card__title--active` |

## 四、Git 工作流

### 4.1 分支策略（Git Flow 简化版）

```
main          ──●──────────────────●──────────────── 生产环境
                │                   ↑
develop       ──●──●──●──●──●──●──●── 开发主线
                    │     │     ↑
feature/*         ──●──●──┘     │   功能分支
                          │     │
hotfix/*                  ──●───┘   紧急修复
```

| 分支 | 命名 | 用途 | 合并目标 |
|------|------|------|----------|
| main | main | 生产环境代码 | — |
| develop | develop | 开发主线 | main（发布时） |
| feature | feature/xxx | 功能开发 | develop |
| hotfix | hotfix/xxx | 紧急修复 | main + develop |

### 4.2 分支命名规范

```
feature/user-auth          # 功能
feature/swing-analysis     # 功能
fix/video-upload-crash     # Bug 修复
hotfix/payment-callback    # 紧急修复
refactor/api-structure     # 重构
docs/api-design            # 文档
```

### 4.3 Commit Message 规范

使用 Conventional Commits：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**type 枚举**：

| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档 |
| style | 代码格式（不影响逻辑） |
| refactor | 重构 |
| perf | 性能优化 |
| test | 测试 |
| chore | 构建/工具变动 |
| ci | CI/CD 配置 |

**示例**：

## 七、日志规范

### 7.1 日志级别使用规范

| 级别 | 使用场景 |
|------|----------|
| DEBUG | 开发调试信息（生产环境关闭） |
| INFO | 关键业务事件（用户登录、分析完成、支付成功） |
| WARNING | 异常但可恢复的情况（重试、降级） |
| ERROR | 业务错误（分析失败、支付失败） |
| CRITICAL | 系统级故障（数据库连接失败、服务不可用） |

### 7.2 日志格式

```json
{
  "timestamp": "2026-04-14T10:35:00.123+08:00",
  "level": "INFO",
  "module": "analysis_service",
  "message": "Swing analysis completed",
  "user_id": "usr_abc123",
  "analysis_id": "ana_def456",
  "duration_ms": 18500,
  "trace_id": "req_xyz789"
}
```

### 7.3 请求追踪

- 每个 HTTP 请求分配唯一 `trace_id`（通过中间件）
- `trace_id` 贯穿日志、异步任务、第三方调用
- 响应头返回 `X-Trace-Id` 便于排查

## 九、安全规范

### 9.1 敏感信息管理

- 所有密钥、密码通过环境变量注入，禁止硬编码
- `.env` 文件加入 `.gitignore`
- 生产环境使用腾讯云密钥管理服务（KMS）
- 日志中禁止输出密钥、Token、密码等敏感信息

### 9.2 接口安全

- 所有接口强制 HTTPS
- JWT Token 签名使用 HS256（MVP）/ RS256（后期）
- 所有用户输入进行 XSS 过滤和 SQL 注入防护（Pydantic + ORM 天然防护）
- 文件上传校验文件头（Magic Bytes），防止恶意文件

### 9.3 数据安全

- 数据库密码使用强密码（16 位以上）
- 数据库连接使用 SSL
- 用户视频数据使用 COS 私有读权限 + 临时签名 URL
- 用户支付信息不在本地存储，全部委托微信支付处理
