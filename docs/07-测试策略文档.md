## 一、测试目标与原则

### 1.1 测试目标

- 确保 MVP 核心流程（视频分析、AI 对话、支付）零阻断性 Bug 上线
- 保证 API 接口契约与文档一致
- 验证 AI 分析准确率达到可发布标准
- 确保数据安全与隐私合规要求落地

### 1.2 测试原则

| 原则 | 说明 |
|------|------|
| 分层覆盖 | 单元 → 集成 → E2E → AI 专项，逐层递进 |
| 自动化优先 | 核心流程必须自动化，手工测试仅用于 UI/UX 和探索性测试 |
| 左移测试 | 开发阶段即编写测试，PR 合并前 CI 必须通过 |
| 数据隔离 | 测试环境与生产环境完全隔离，使用独立数据库和模拟服务 |


**前端（TypeScript / Jest + React Testing Library）**

| 类别 | 覆盖范围 | 覆盖率目标 |
|------|----------|-----------|
| 状态管理 | Zustand Store：用户状态、分析状态、配额状态 | ≥ 80% |
| 工具函数 | 格式化、校验、时间处理 | ≥ 90% |
| 组件 | 核心交互组件（非纯 UI） | ≥ 60% |

#### 前端测试基础设施约定（W9 落地）

> 配置入口：`client/jest.config.cjs`、`client/babel.config.test.cjs`、`client/jest.setup.cjs`、`client/jest.polyfills.cjs`、`client/src/__mocks__/`。

- **运行器**：`Jest 29 + jest-environment-jsdom`；编译器 `babel-jest`（与 Taro 自身 babel 编译共用 preset 语义），不引入 `ts-jest` 以避免双套 TS 编译漂移。
- **覆盖范围**（与 `collectCoverageFrom` 一致）：
  - `src/services/**`（含 `request.ts` 错误统一封装）
  - `src/utils/**`（含 `storage` / `privacy` / `wxDomainMessages` 等端无关纯函数）
  - `src/store/**`（zustand store 的 actions / 错误映射）
  - `src/components/**`（端无关的 React 组件 DOM 层；`*.rn.tsx` 由 W10 RN smoke 覆盖）
  - `src/constants/**`
- **不纳入**：`src/pages/**`（含 Taro 路由生命周期，端耦合重）、`src/adapters/**`（按端分叉，单元测试无法跨端验证；交由 RN bundle + 真机 smoke 兜底）。
- **测试目录**：`src/<module>/__tests__/<filename>.test.{ts,tsx}` 与被测文件同级，便于代码评审时一眼对照。
- **Taro 桩**：`src/__mocks__/tarojs.ts`（@tarojs/taro 整包）+ `src/__mocks__/tarojs-components.tsx`（@tarojs/components → HTML 标签）。新增 Taro API 调用前，先在桩里补齐对应 `jest.fn`，再写测试，避免在每个 `*.test.ts` 里手工 mock 同一个 API。
- **首批锁定的关键不变式**（修改这些行为务必先看测试）：
  - `userStore.bootstrap` 弱网时**不踢用户**（5xx/network 保留 token + cachedUser）。
  - `storage.clearAuthSession` 只清账号身份，**保留** `agreed_terms`、`analysis_guide_seen`（合规体验底线）。
  - `request.ts` 401 + `code=40104` 必须按业务错误处理，**不**清 token。
  - `paymentService.createOrder` / `chatService.sendMessage` 默认 `silent=true`，UI 自行编排错误文案。
  - `userService.wechatLogin` 端分叉：weapp `/auth/wechat-login`，rn `/auth/wechat-open-login`。
  - `chatStore.submitMessageSync` 失败时**必须回滚** user + assistant 占位气泡。
- **本地命令**：
  - `cd client && pnpm install`（首次或新加依赖后必须）
  - `make client-test`：交互式
  - `make client-test-watch`：监听
  - `make client-test-coverage`：带覆盖率与阈值（见 `coverageThreshold`）
  - `make client-test-ci`：CI 模式（单 worker / silent）
- **CI**：`.github/workflows/client-jest.yml` 与 `client-rn-check` 并行；任一红 → PR 卡红。
- **新增测试时**：优先复用上述桩；如确需引入新依赖，先在 PR 描述里说明 bundle 影响（Jest 测试依赖应限定为 `devDependencies`）。

### 2.2 集成测试

**API 集成测试（pytest + httpx）**

针对每个 API 端点编写集成测试，使用真实数据库（测试库）+ Mock 外部服务。

| 模块 | 测试要点 |
|------|----------|
| 用户认证 | 微信登录全流程（Mock 微信 API）、Token 刷新、Token 过期 |
| 挥杆分析 | 上传凭证获取 → 创建分析 → 状态轮询 → 报告获取（Mock AI 引擎） |
| AI 对话 | 创建会话 → 发送消息 → SSE 流式响应 → 对话历史（Mock LLM） |
| 支付 | 创建订单 → 模拟支付回调 → 会员状态更新 → 自动续费取消 |
| 邀请裂变 | 邀请码绑定 → 有效邀请判定 → 奖励发放 → 防刷拦截 |
| 配额管理 | 月度配额扣减/退回 → 日度配额重置 → 奖励配额叠加 |
| 账号注销 | 申请 → 冷静期取消 → 到期执行（Mock 定时任务） |

**Mock 策略**：

| 外部服务 | Mock 方式 |
|----------|----------|
| 微信 code2session | pytest fixture 返回固定 openid |
| 微信支付 | 模拟支付回调通知 |
| 腾讯云 COS | 使用 MinIO 本地替代 |
| AI 分析引擎 | 返回预设分析结果 JSON |
| LLM（DeepSeek/Qwen） | 返回固定文本流 |

```
# 测试命令
make test-integration              # 运行全部集成测试
docker-compose -f docker-compose.test.yml up -d  # 启动测试依赖
```

### 2.4 AI 模型专项测试

**挥杆分析准确度测试**

| 测试维度 | 方法 | 通过标准 |
|----------|------|----------|
| 姿态估计精度 | 与人工标注对比 PCK@0.2 | ≥ 80% |
| 阶段分割准确率 | 与人工标注的 IoU 对比 | ≥ 75% |
| 评分一致性 | 同一视频多次分析评分偏差 | ≤ ±3 分 |
| 问题诊断准确率 | 与专业教练标注对比 | 精确率 ≥ 80%，召回率 ≥ 70% |
| 推荐相关性 | 训练建议与诊断问题的匹配度 | 人工评审通过率 ≥ 85% |

**测试数据集**：

| 数据集 | 规模 | 用途 |
|--------|------|------|
| 基准测试集 | 100 个标注视频 | 模型准确度评估 |
| 边界测试集 | 30 个异常视频 | 鲁棒性测试（光线差/遮挡/非标角度） |
| 回归测试集 | 50 个视频 + 固定预期 | 模型更新时防回归 |

**AI 对话质量测试**

| 测试维度 | 方法 | 通过标准 |
|----------|------|----------|
| 回答相关性 | 100 个预设问题 + 人工评分 | 平均 ≥ 4/5 |
| 上下文理解 | 20 组多轮对话场景 | 上下文保持率 ≥ 90% |
| 安全过滤 | 50 个越界/敏感问题 | 拦截率 100% |
| 幻觉检测 | 30 个事实性问题 | 错误回答率 < 5% |
| 响应时间 | 首 Token 延迟统计 | P95 < 2 秒 |

## 四、安全测试

### 4.1 必测安全项

| 类别 | 检查项 |
|------|--------|
| 认证 | Token 伪造检测、过期 Token 拒绝、缺失 Token 拦截 |
| 授权 | 用户 A 不能访问用户 B 的分析报告/对话/订单 |
| 输入校验 | SQL 注入、XSS、路径遍历、文件上传恶意文件 |
| 限流 | 验证各接口限流策略生效 |
| 数据加密 | 传输 TLS 1.3、OpenID 存储加密验证 |
| 隐私 | 邀请记录中昵称脱敏、分享卡片无敏感信息 |
| 视频安全 | 签名 URL 过期验证、非授权访问拦截 |

### 4.2 合规验证

| 检查项 | 验证方式 |
|--------|----------|
| 隐私政策弹窗 | 新用户首次进入必须展示 |
| 数据授权流程 | 逐项授权，可选授权不影响核心功能 |
| 数据导出 | 用户可导出个人数据 |
| 账号注销 | 注销后数据正确清理 |
| LLM 数据脱敏 | 发往 LLM 的数据不包含用户唯一标识 |

## 六、测试环境

### 6.1 环境架构

| 环境 | 用途 | 数据 | 外部服务 |
|------|------|------|----------|
| local | 开发自测 | Docker Compose（PG + Redis + MinIO） | 全部 Mock |
| test | CI 自动化测试 | 独立测试数据库，每次运行前重置 | 全部 Mock |
| staging | 预发布联调 | 脱敏数据 | 微信沙箱 + 真实 COS（测试桶） |
| production | 线上 | 真实数据 | 真实服务 |

### 6.2 测试数据管理

- **Fixture 管理**：使用 `pytest fixtures` + `factory_boy` 生成测试数据
- **测试数据库**：每个测试用例独立事务，完成后自动回滚
- **种子数据**：`drills` 和 `daily_tips` 表使用固定种子数据
- **视频测试文件**：测试用视频存放在 `tests/fixtures/videos/` 目录

## 七、CI/CD 集成

### 7.1 PR 合并门禁

```yaml
# .github/workflows/test.yml 中的必过检查
必须通过:
  - Ruff lint（后端代码风格）
  - ESLint + Prettier（前端代码风格）
  - 单元测试 100% 通过
  - 集成测试 100% 通过
  - 单元测试覆盖率 ≥ 70%（新代码 ≥ 80%）
  - 类型检查通过（mypy / tsc）

建议通过:
  - AI 模型测试（仅模型相关 PR）
  - 性能基准测试（仅关键路径 PR）
```

### 7.2 发布前测试流程

```
代码冻结
  → staging 环境部署
  → 自动化回归测试（E2E）
  → 性能测试（负载 + 重点接口）
  → 安全扫描
  → 兼容性测试（人工 + 核心设备）
  → 产品验收
  → 灰度发布（5% → 20% → 50% → 100%）
```

## 八、Bug 管理规范

### 8.1 Bug 严重等级

| 等级 | 定义 | 修复时限 | 示例 |
|------|------|----------|------|
| P0 - 致命 | 核心功能完全不可用 | 4 小时内 | 登录失败、支付异常、数据泄露 |
| P1 - 严重 | 核心功能部分异常 | 24 小时内 | 分析结果错误、对话不可用、配额计算错误 |
| P2 - 一般 | 非核心功能异常或 UI 缺陷 | 当前迭代内 | 分享卡片布局异常、通知延迟 |
| P3 - 轻微 | 体验优化类 | 排期修复 | 文案错别字、非核心页面样式 |
