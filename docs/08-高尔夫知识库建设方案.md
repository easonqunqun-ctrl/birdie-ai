## 一、知识库定位

### 1.1 核心目标

为 AI 对话教练提供高质量、结构化的高尔夫专业知识作为 RAG（检索增强生成）的知识源，使 AI 回答准确、专业且具有中国高尔夫语境的本地化特色。

### 1.2 能力边界

| 能力范围 | MVP 覆盖 | 后续迭代 |
|----------|---------|----------|
| 挥杆技术（全挥杆、短杆、推杆） | ✅ | 扩充细分 |
| 常见错误诊断与纠正 | ✅ | 更多错误类型 |
| 训练方法与练习动作 | ✅ | 视频教程 |
| 高尔夫基础规则与礼仪 | ✅ | — |
| 装备基础知识 | ✅ 通用建议 | 品牌/型号推荐 |
| 球场策略 | ✅ 通用策略 | 特定球场攻略 |
| 身体素质与热身 | ✅ 基础 | 体能训练方案 |
| 运动心理 | ✅ 基础 | 专项心理辅导 |
| 青少年教学 | ❌ | ✅ |
| 比赛策略 | ❌ | ✅ |

## 三、内容来源与采集

### 3.1 MVP 阶段内容来源

| 来源 | 内容类型 | 预估条目 | 版权处理 |
|------|----------|---------|----------|
| PGA/LPGA 公开教学资料 | 挥杆原理、训练方法 | 80 条 | 提炼改写，注明参考 |
| R&A 高尔夫规则手册 | 规则、判罚 | 50 条 | 原文引用（公开文件） |
| 高尔夫教学书籍摘要 | 技术细节 | 60 条 | 人工提炼改写 |
| 高尔夫专业媒体文章 | 技巧提示 | 40 条 | 人工改写 |
| 专业教练原创内容 | 纠正方法、训练心得 | 30 条 | 原创/约稿 |
| 高尔夫术语库 | 中英文术语对照 | 200+ 条 | 整理编撰 |
| 训练动作库 | drill 详细说明 | 20+ 条 | 原创 |
| **合计** | | **~480 条** | |

### 3.2 内容采集流程

```
1. 选题规划 → 按知识体系框架确定待采集条目
2. 内容采集 → 从原始来源提取相关段落
3. 内容改写 → 用通俗易懂的中文重新编写
4. 结构化 → 按标准格式整理，添加标签和关联
5. 专业审核 → 高尔夫教练或专业人士审核准确性
6. 入库标注 → 导入向量数据库，标注元数据
7. 质量验证 → 通过 QA 测试验证检索效果
```

### 3.3 内容质量标准

| 维度 | 标准 |
|------|------|
| 准确性 | 技术描述必须经过专业教练审核，不允许有误导性内容 |
| 通俗性 | 使用日常中文表述，技术术语附带中英文对照和解释 |
| 完整性 | 每个问题包含"现象描述 → 原因分析 → 纠正方法 → 练习建议" |
| 安全性 | 涉及身体动作的内容须附安全提示，避免运动损伤 |
| 本地化 | 适应中国高尔夫爱好者的语境（练习场 vs 球场、常用装备品牌等） |

## 四、向量化与索引

### 4.1 文本分块策略

| 参数 | 值 | 说明 |
|------|------|------|
| 分块方式 | 语义分块（Semantic Chunking） | 按段落/知识点自然分割 |
| 目标块大小 | 300-500 tokens | 兼顾语义完整性和检索粒度 |
| 重叠窗口 | 50 tokens | 跨块语义连续性 |
| 块最大长度 | 800 tokens | 硬性上限 |
| 元数据保留 | title, category, tags, difficulty, related_drills | 用于过滤和重排 |

### 4.2 Embedding 模型

| 方案 | 模型 | 维度 | 优势 | 适用场景 |
|------|------|------|------|----------|
| 首选 | bge-large-zh-v1.5 | 1024 | 中文语义理解强，开源免费 | MVP 阶段 |
| 备选 | text-embedding-3-small | 1536 | 多语言能力强 | 如需中英文混合检索 |

### 4.3 向量数据库

| 项目 | 选型 |
|------|------|
| 数据库 | Qdrant |
| 部署方式 | Docker 单机（MVP）→ 集群（扩展期） |
| Collection 设计 | `golf_knowledge`（主知识库）、`drills`（训练动作专库） |
| 索引类型 | HNSW |
| 距离度量 | Cosine Similarity |

**Collection Schema**：

```json
{
  "collection_name": "golf_knowledge",
  "vectors": {
    "size": 1024,
    "distance": "Cosine"
  },
  "payload_schema": {
    "title": "keyword",
    "category": "keyword",
    "tags": "keyword",
    "difficulty": "keyword",
    "content": "text",
    "source": "keyword"
  }
}
```

### 4.4 检索策略

```
用户问题
  ↓
意图识别（分类器）
  ↓
Query 改写（关键词提取 + 问题扩展）
  ↓
混合检索：
  ├── 向量检索（Cosine Top-10）
  └── 关键词检索（BM25 Top-10）
  ↓
结果融合（RRF - Reciprocal Rank Fusion）
  ↓
重排序（基于 category/difficulty 匹配度加权）
  ↓
取 Top-3 作为 LLM 上下文
```

**检索质量指标**：

| 指标 | 目标值 |
|------|--------|
| Top-3 命中率 | ≥ 85%（在 100 个标准测试问题上） |
| Top-5 命中率 | ≥ 92% |
| 平均检索延迟 | < 100ms |
| 无关结果混入率 | < 5% |

## 五、知识库运维

### 5.1 更新机制

| 类型 | 频率 | 流程 |
|------|------|------|
| 新增内容 | 每周 | 运营提交 → 教练审核 → 向量化入库 |
| 内容修正 | 随时 | Bug/反馈触发 → 修正 → 重新向量化 |
| 规则更新 | 每年 | R&A 规则变更时批量更新 |
| 全量重建 | 按需 | Embedding 模型升级时全量重建索引 |

### 5.2 质量监控

| 监控项 | 方法 |
|--------|------|
| 检索失败 | 监控"未找到相关知识"的对话比例，目标 < 10% |
| 用户反馈 | AI 回答后的"有帮助/没帮助"按钮，跟踪满意度 |
| 知识空白 | 定期分析未命中的用户问题，识别知识缺口 |
| 过期内容 | 季度审查，标记过期条目 |

### 5.3 知识库管理工具

MVP 阶段使用脚本管理，后续可建设管理后台：

```
# 知识库管理命令
make kb-import FILE=data/knowledge/swing.json     # 导入知识条目
make kb-rebuild                                    # 全量重建向量索引
make kb-search QUERY="怎么纠正右曲球"              # 测试检索效果
make kb-stats                                      # 统计知识库概况
make kb-validate                                   # 校验条目格式和完整性
```

## 七、每日小贴士库

### 7.1 内容规划

MVP 阶段至少准备 60 条小贴士，确保 2 个月内不重复：

| 类别 | 数量 | 示例 |
|------|------|------|
| 技术提示 | 20 条 | "练球前的 5 分钟热身，可以让你的挥杆更稳定。" |
| 心态建设 | 10 条 | "每个职业球手都经历过瓶颈期，坚持练习就会突破。" |
| 趣味知识 | 10 条 | "职业球手的上杆和下杆时间比例通常是 3:1。" |
| 装备常识 | 10 条 | "定期检查握把磨损程度，磨损的握把会影响挥杆稳定性。" |
| 规则小知识 | 10 条 | "球在果岭上可以标记位置后拿起来清洁。" |

### 7.2 展示策略

- 每天随机展示 1 条（同一天内固定不变）
- 30 天内不重复展示
- 可根据用户的分析结果进行智能推荐（后续迭代）

## 八、实施计划

| 阶段 | 时间 | 交付物 |
