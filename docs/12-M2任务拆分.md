## 技术决策（5 条，影响所有子任务）

1. **对象存储**：W3-W4 **仅用 MinIO**，客户端直传（wx.uploadFile + Form + 本地签名凭证）；COS 真机适配放 W8。后端提供 `MinioStorageClient` 抽象，未来可无痛切 COS。
2. **客户端上传**：**直传**对象存储 —— 后端 `POST /v1/analyses/upload-token` 返回 MinIO 的**预签名 POST policy**（`x-amz-*` 字段），客户端 `wx.uploadFile` 直接 PUT/POST。不走后端代理，避免 100MB 视频过后端。
3. **异步编排**：**Celery + Redis broker**，与生产一致；`docker-compose.yml` 新增 `celery-worker` 容器。开发期 worker 热重载用 `celery -A app.celery_app worker --loglevel=info`。
4. **报告页范围**：完整版 —— 视频回放、骨骼叠加占位、六维雷达图、问题诊断、训练建议、阶段色条、0.25x/0.5x/1x 速度切换。`skeleton_video_url` 字段在 mock 模式下指向原视频（或一个预先生成的占位 mp4），W6 真实化时无需改 UI。
5. **分享/对比延后**：`POST /v1/analyses/{id}/share-card` 和"对比历史"放 M5/W7；当前报告页底部"对比历史"按钮保留占位文案，点击提示"W7 上线"。

## T2 · 后端 + AI Engine：Celery 工作流 + Mock 分析落库（P0）—— ✅ 已完成

**目标**：让 T1 创建的 `pending` 任务真正跑起来 —— Celery 接单 → 调 AI Engine `/analyze` → 结果入 DB；失败退配额。

### ✅ 交付清单（2026-04-20）

- **Celery 实例**：`backend/app/celery_app.py`。broker/result 分别放 Redis DB 1 / DB 2（业务 cache 继续用 DB 0），`task_time_limit=300s`，`worker_prefetch_multiplier=1`。
- **AI Engine 客户端**：`backend/app/integrations/ai_engine.py`。`AIEngineClient.analyze(...)` 返回 dict（不用 Pydantic 模型反序列化，避免后端/ai_engine schema 命名同但语义细微差异），超时原样抛给上层。
- **任务主体**：`backend/app/tasks/analysis_tasks.py`。
  - `run_swing_analysis` 是 celery sync task，body 用 **独立线程 + `asyncio.run()`** 跑 `_run_swing_analysis_async`，既能在 worker 独立进程跑，也能在 eager 模式/测试下绕开 "running loop" 冲突。
  - `_run_swing_analysis_async` 内部自己做 3 次重试（1 次主调 + 2 次重试、1s/2s 退避），**不依赖 Celery autoretry**，测试可精准控制。
  - `_mark_processing / _mark_failed / _mark_completed` 分离，幂等（二次消费 terminal 任务直接 return，不重写 issues/不加扣配额）。
  - 失败退配额通过新加的 `quota_service.refund_analysis_quota_by_user_month(db, user_id=..., quota_month=...)` 实现；月份按 `created_at` 的北京时间算，跨月也不会退错月。
- **调度接入**：`analysis_service.create_analysis` 末尾调 `_dispatch_analysis_task(analysis_id)`，后者再 `run_swing_analysis.delay(...)`。抽出单独函数是为了测试里 `monkeypatch` 拦截（不依赖 celery broker 就能跑完 API 链路）。
- **docker-compose**：新增 `celery-worker` 服务（复用 backend 镜像；`command: uv run celery -A app.celery_app worker --loglevel=info --concurrency=2`）。Makefile 加 `make backend-celery-logs / backend-celery-shell`。
- **错误码修正**：AI 引擎不可达原计划用 50201，但 50201 已被"微信服务异常"占用 → 改用 **50104 "AI 引擎不可达"**，`docs/02 §1.4` 错误码表已同步登记。
- **测试**：`tests/test_analyses_e2e.py` 共 6 条，全部绿：
  1. happy path：worker 消费后 `/status=completed`、`/report` 返回完整六维 + issues + recommendations + skeleton/thumbnail。
  2. `engine_failed`：ai_engine 返回 `status=failed` → 落库 failed、`error.quota_refunded=True`、配额回到调用前，ai_engine 仅被调一次。
  3. `timeout`：3 次全超时 → 终态 failed（code=50104）、配额恢复、`call_count==3`。
  4. `flaky`：前 2 次超时第 3 次成功 → 最终 completed，配额**不**退回。
  5. 多任务独立：同一用户连跑 2 条都完整落库、互不干扰。
  6. worker 幂等：已 completed 的 analysis 再消费一次，issues 数量不变、`analyzed_at` 不被覆盖。
- **手跑 smoke**：`bash docs/api-examples/analyses-lifecycle.sh` 扩展为"轮询 status 直到 completed/failed（上限 60s）"并请求 `/report`。实测：`pending → processing(preprocessing, 10%) → completed`，4 秒全部跑完，`Task xiaoniao.run_swing_analysis[...] succeeded in 3.42s`（见 `docker compose logs celery-worker`）。

### 未包含 / 显式延后

- `ai_engine/app/sample_fixture.py`（示例视频固定返回）**未做**，T6 再补。
- `ai_engine/app/schemas.py` 的 `IssueItem.key_frame_url` 字段**未加**（ai_engine 里仍是只返回 `key_frame_timestamp`）。后端已预留字段 `AnalysisIssue.key_frame_url`，写入 `None`；客户端 T5 用 `onerror` 兜底即可。W6 真实 pipeline 引入时再补这个字段。

### 原设计（作为参考保留）

**范围**：

**范围**：
- `docker-compose.yml` 新增 `celery-worker` 服务（与 backend 同镜像，入口 `celery -A app.celery_app worker --loglevel=info --concurrency=2`）

- **配额退回语义**：`refund_analysis_quota` 将 `used -= 1`，若之前已是会员/赠送，不做区分；失败的 analysis 仍保留记录（`status=failed`，仅从列表页默认过滤）
