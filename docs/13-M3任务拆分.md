## 三、技术要点 & 风险提前登记

### 3.1 LLM 选型与抽象

- **抽象层**：`app/integrations/llm.py::LLMClient`，走 **OpenAI 兼容 API** —— 这样 DeepSeek / 通义千问 / Moonshot / 本地 vLLM 都能零改切换。
- **默认适配**：MVP 期默认跑 **DeepSeek**（性价比 + 中文好 + OpenAI 兼容最稳）。
- **配置**：`.env` 读取 `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL_NAME` / `LLM_TEMPERATURE`。
- **Mock 模式**：`LLM_API_KEY` 为空时自动走 `FakeLLM`，按字符吐流，便于 CI / 本地开发 / 离线演示。
- **超时**：首 token 10s，总生成 60s（对齐 MVP §8 "首 token < 2s" + 余量）。

### 3.2 SSE 与小程序的"不兼容"⚠️

**后端侧**：FastAPI 用 `StreamingResponse(media_type="text/event-stream")`，每行 `event: xxx\ndata: {...}\n\n`。

**前端侧风险（必读）**：
- `wx.request` **不支持标准 EventSource**。
- 解决路径：
  1. **首选**：`wx.request({ enableChunked: true })`（基础库 ≥ 2.20.1）+ `onChunkReceived` 接收 chunk，本地缓冲拼成行，手动解析 SSE 事件。
  2. 备选：在真机上若 `enableChunked` 有兼容问题，降级到 **"伪流式"** —— 服务端把 SSE 响应同时写入 Redis，客户端短轮询 `GET /messages/{id}/stream?cursor=N` 拉增量（T4 内收敛方案）。
- 本文档 T2 产出标准 SSE 接口，T4 让前端优先走 `enableChunked`，**实测不过再降级**，降级逻辑的代码改动控制在前端 + 一个新的 GET 增量接口。

### 3.3 上下文构造

每次调 LLM 的 `messages` 数组由 **三段** 组成（顺序固定）：

```
[
  {role: "system", content: <system_prompt>},
  ...<最近 N 轮历史消息，N ≤ 20，从 chat_messages 倒序取后正序拼>,
  {role: "user", content: <本次用户消息>}
]
```

**system_prompt 的动态部分**（每次请求都刷新）：
- 用户昵称、球龄、最近常用球杆（从 `user_profile`）
- **最近 3 次** `swing_analyses` 摘要（overall_score / 弱项 / 前 2 条 high-severity issues）
- 硬约束："只回答高尔夫话题"、"不超 300 字"、"医疗问题引导去看医生"
- 版本号：`system_prompt_version = "v1"`（落库到 `chat_sessions.system_prompt_version` —— docs/03 §3.5 有定义；**实现偏差**：当前 `ChatSession` 模型缺此字段，T1 补一条小迁移或作为 future work 登记）

### 3.4 配额与速率限制

- **每日轮次**：`chat_quotas(user_id, quota_date)`，`total` 默认 5（免费）/ `-1`（会员无限）。一"轮" = 1 条 user message。message_start 时预扣，失败退回。
- **速率限制**：每用户 20 次/分钟。走 Redis `INCR chat_rate:{user_id}:{unix_minute}` + `EXPIRE 120`，满 20 拒绝（429 + code=40009）。
- **错误码**：
  - `40007` 今日对话次数不足（文档已有）
  - `40009` 请求过于频繁（speed limit，本期新增，docs/02 §1.4 需补登记）
  - `50106` AI 对话服务异常（M3-T1 修正：原 `50105` 已被 M2-T6 "视频画质不足"占用，对话服务异常改用 50106）

**关键实现注意**：
1. **SSE 开流前必须做所有 4xx**：`StreamingResponse` 一旦开始写 body，FastAPI 异常处理器就失效（会抛 "response already started"）。所以 `prepare_turn` 同步跑，异步生成器只做 LLM 调用
2. **速率限制必须在 SSE 开流前**：放在 `check_chat_send_rate` 里，触发 40009 时 SSE 头都还没发
3. **placeholder key 识别**：`.env.example` 里 `LLM_API_KEY=sk-placeholder` 被匹配到自动回退 Fake，避免本地联调误打外网
4. **conftest 的 monkeypatch 双重绑定**：service 模块用 `from ... import get_llm_client` 把 name 绑定到局部，必须同时在 llm 模块和 service 模块两处 patch，否则测试看似"注入了 Fake 但实际走 DeepSeek"

- **业务 error vs 传输 error 必须分流**：后端 `event: error` 发出时 user_msg 已落库、配额已退；传输层断开（abort / 超时 / 502）后端可能根本没收到或没来得及退配额。两条路径的 UI 提示 + store 状态**完全不同**（前者 assistant 保留部分内容，后者 assistant 兜底为"网络中断"），两个回调名字也刻意区分

**关键设计选择 & 踩坑**：

- **sample 报告不带 analysis_id**：如果把 `sample` 当真实 id 发给 `POST /chat/sessions`，后端 `quota_service` 会试图查 `swing_analyses.id = 'sample'` 行并返回 404（示例数据只在 `GET /analyses/sample` 特殊路径下硬编码）。所以跳转时**显式 drop** 掉 analysis_id，让 chat 走通用 system prompt
- **prefill 单次生效用 ref 而非 state**：state 改写会触发额外 render；ref 纯"做一次标记"没有渲染成本，也不会因热重载重置（热重载时 ref 被保留，符合预期）
- **"查看原报告"优先 navigateBack**：报告页 `report.tsx` 的 Video 组件带播放进度；`navigateTo` 重开会丢进度。只有在用户从首页入口进 chat 后才 fallback `navigateTo`
- **首页 CTA 用 `chat_remaining_today >= 0` 判分支**：后端用 `-1` 表示会员无限次；UI 不能直接 `?? 0`（会把"无限"误展示成"今日剩余 0 次"）
- **所有 tsc/lint/build 速度已恢复**：T4 那次 type-check 跑了 133s 是冷缓存 IO 等待（`Parse 49s`）；重跑后 1.5s 完成。后续任务默认按"编译在秒级"预期
