## 一、里程碑定位

M2 已经把**分析链路的壳**搭好了：上传 → Celery → HTTP 调 `ai_engine` → 报告落库 → 前端渲染。W6 不重构链路，**只替换 `ai_engine` 里"算分析结果"那一坨**：把 `mock_pipeline.run_mock_analysis` 的随机数换成基于 MediaPipe Pose 的真实推理。

所以 W6 的特点：
- **backend 基本不动**：API / Celery / DB schema 已就位；仅 `AI_ENGINE_TIMEOUT` 调参 + stage 推进细化
- **前端基本不动**：`report.tsx` 字段已兼容；仅 `waiting.tsx` 去掉"stage 时间模拟"垫层

- **Docker 加依赖**：ffmpeg + mediapipe（slim 镜像 + pip wheel 即可，不上 GPU）

## 三、技术选型

| 维度 | 方案 | 理由 |
|------|------|------|
| 姿态估计 | **MediaPipe Pose**（pip `mediapipe>=0.10`，自带模型） | docs/05 指定；CPU 推理 10s 视频 ~3-8s；无需额外模型分发 |
| 转码/抽帧 | **ffmpeg CLI**（subprocess） | 转码 H.264 / 30fps / 720p 短边；稳定输出喂给 OpenCV |
| 读帧 | OpenCV `VideoCapture` | 与 MediaPipe 天然配合（BGR→RGB）；同时用于稳像/抖动指标 |
| 阶段分割 | **Rule-based**（手腕速度峰 + 杆头 Y 极值 + 肩髋角） | docs/05 §2.4 明确 MVP 用启发式；误差 ±3 帧@30fps 可接受 |
| 特征 | 纯 NumPy 算 **15 个核心特征**（docs/05 §2.5） | 无外部依赖，可单元测试；CPU 开销可忽略 |
| 评分 | 阶段内特征加权 + 阶段全局加权（docs/05 §2.6，**与 mock_pipeline.PHASE_WEIGHTS 一致**） | 保持与 W3-W4 已上线的前端雷达图兼容 |
| 骨骼时序落地 | **parquet 到 MinIO**，`skeleton_data_url` 指向 | 30s@30fps × 33×3 float32 ≈ 350KB；parquet 比 JSON 省 3-5 倍 |
| 关键帧图 | OpenCV `imwrite` JPG → MinIO | issue 命中那一帧截出来，`key_frame_url` 指向；前端已用 `||thumbnail_url` 兜底 |
| 叠加视频 | OpenCV 绘点连线 + ffmpeg 重封装 H.264 | 播放器直接 `<video>` 播；不需额外解码器 |
| 容器基础镜像 | `python:3.11-slim-bookworm` + `apt install ffmpeg libgl1 libglib2.0-0` | mediapipe wheel 在 bookworm 上 OK；体积 ~600MB 可接受 |

## 四、任务拆分（T1-T6）

### T1 · 预处理 + MediaPipe 姿态估计（P0）✅

**目标**：吃一段 mp4，吐 `(num_frames, 33, 3)` 关键点 + 帧级置信度 + `quality_score/warnings`。

**范围**：
- `ai_engine/app/pipeline/preprocess.py`：ffmpeg 子进程转码 30fps / 720p 短边 / H.264 `yuv420p`；OpenCV 读帧算**清晰度**（拉普拉斯方差）和**稳像**（帧间平均位移）
- `ai_engine/app/pipeline/pose.py`：MediaPipe Pose `static_image_mode=False, model_complexity=1, min_detection_confidence=0.5`；逐帧推理；输出 `np.ndarray[F, 33, 3]`（xyz 归一化到 [0,1]）+ `confidence[F]` + `valid_mask[F]`
- **质量门**：若 `valid_mask.mean() < 0.7` 或低清晰度帧比例 > 30% → 抛 `PoorQualityError` → `/analyze` 返回 50102（映射 docs/02 §1.4）
- **无人物/无挥杆**：若所有帧无检出 → 50103

**验收**：
- 单元测试：至少 3 段 fixture 视频 × 不同角度 → 关键点维度正确 + `confidence.mean() > 0.6`
- 性能测试：10s 视频端到端 < 8s（CPU 8 核基线）

**依赖/踩坑预警**：
- MediaPipe 0.10+ 的 Pose API 与 0.9 不同（`Solutions` → `Tasks` API 迁移）；锁定一个版本并写注释
- ffmpeg 子进程错误流要捕获，避免 subprocess hang（`subprocess.run(..., timeout=60, capture_output=True)`）
- MediaPipe 在某些 Linux 上对 GLIBC 版本有要求；bookworm 是安全选择，alpine 不行

**影响文件**：
- **新增**：`ai_engine/app/pipeline/{__init__,preprocess,pose}.py`、`ai_engine/app/errors.py`、`ai_engine/tests/fixtures/*.mp4`、`ai_engine/tests/test_pose.py`

### T2 · 阶段分割 + 15 特征 + 评分 + 诊断（P0）✅

**目标**：吃 T1 输出的关键点序列，吐 `phase_timestamps / phase_scores / overall_score / issues[] / recommendations[]` —— **和 mock 同 schema**。

**范围**：
- `ai_engine/app/pipeline/phases.py`：用手腕 Y 轴速度峰值（上杆顶点 / 击球时刻）+ 肩部旋转角（setup/follow_through 边界）识别六阶段；返回 `{phase: {start, end, peak?}}`
- `ai_engine/app/pipeline/features.py`：实现 docs/05 §2.5 的 15 个核心特征（脊柱角 / 肩髋分离 / 头高变化 / 手腕释放时机 / 杆头路径 / ...）；每个特征给 `{value, normal_range, deviation}` 三元组
- `ai_engine/app/pipeline/scoring.py`：
  - 阶段内：15 特征按所属阶段加权 → `phase_scores[phase] = {score: int, feedback: str, is_weakest: bool}`
  - 全局：`overall_score = sum(phase_scores[p].score * PHASE_WEIGHT[p])`，权重与 `mock_pipeline.PHASE_WEIGHTS` 完全相同
- `ai_engine/app/pipeline/diagnose.py`：基于特征阈值生成 issues；`severity` 规则 = 偏离标准差 × 影响系数；`key_frame_timestamp` 取该 issue 对应阶段的 `peak` 时刻
- `ai_engine/app/pipeline/recommend.py`：按本文件**附录 A** issue→drill 映射表挑 drill_id；每次分析**最多推 3 条**（docs/01 §4.3 对齐）

**验收**：
- 集成测试：fixture 视频 → Pydantic `AnalyzeResult` 校验通过；`phase_timestamps` 六阶段非空；`overall_score ∈ [0, 100]`
- **与 mock 兼容性测**：确保真实输出**只增不改**字段（如果 mock 有 `duration_ms: 2800`，real 也必须有）
- 单元测试：至少 5 种"人造关键点序列"覆盖每个 issue type 的阈值

**依赖/踩坑预警**：
- 15 特征里有几个（比如脊柱角）需要"相机角度校正"，MVP 期**仅 face_on**做精确判断，down_the_line 用降级逻辑（docs/05 §2.5 已注明）
- `is_weakest` 只能有一个阶段；如果有平票，选排序靠前那个
- **issue 去重**：如果 15 特征里有两条都命中 `casting`，合并成一条 issue（用 severity 最高的文案）

**影响文件**：
- **新增**：`ai_engine/app/pipeline/{phases,features,scoring,diagnose,recommend}.py`、`ai_engine/app/constants/issue_drill_map.py`、`ai_engine/tests/test_*.py`
- **改**：`client/src/constants/drillLibrary.ts` 扩充到 15 条 drill（与附录 A 对齐）

### T3 · 骨骼叠加视频 + 关键帧图 + 骨骼时序落 MinIO（P0）✅

**目标**：把 T1-T2 的产物产出三类文件，上传 MinIO，URL 回填 `AnalyzeResult`。

**范围**：
- `ai_engine/app/pipeline/visualize.py`：
  - **骨骼叠加视频**：OpenCV 在每帧画 MediaPipe 的 33 点 + 连线（MediaPipe 自带 `POSE_CONNECTIONS`），再加杆头轨迹（后续增强可选）→ ffmpeg 合成 H.264 → 上传 `skeleton/<analysis_id>.mp4`
  - **关键帧图**：每条 issue 的 `key_frame_timestamp` 抽帧 → JPG（压缩质量 85）→ 上传 `keyframes/<analysis_id>/<issue_type>.jpg` → `issues[i].key_frame_url`
  - **骨骼时序**：`(num_frames, 33, 3) + confidence` 合并成 parquet → 上传 `skeleton_data/<analysis_id>.parquet` → `skeleton_data_url`
- `ai_engine/app/storage.py`：对接现有 `MINIO_*` 配置的轻量 S3 客户端（boto3 已在 backend 依赖里，ai_engine 装一份）

**验收**：
- 产物可通过预签名 URL 下载 + 正常播放/查看
- 总上传耗时 < 3s（三个文件串行）
- 对象命名按分析 id 分目录，便于清理

**依赖/踩坑预警**：
- MinIO bucket 复用 backend 的还是 ai_engine 独立一套？**倾向复用**，少一套凭证；T3 开工前先确认 `MINIO_BUCKET_VIDEO` 有没有 path 隔离
- parquet 需要 pyarrow，装包约 40MB，可接受
- MediaPipe 的 `POSE_CONNECTIONS` 是 0-indexed 坐标对；画的时候 xy 要反归一化 × 视频宽高

**影响文件**：
- **新增**：`ai_engine/app/pipeline/visualize.py`、`ai_engine/app/storage.py`
- **改**：`ai_engine/pyproject.toml`（+ mediapipe, pyarrow, boto3, opencv-python-headless）

### T4 · 后端/Celery/前端联调 + 错误路径（P0）✅

**目标**：mock 关掉，真实引擎接上，错误码对齐，等待页去假动画。

**范围**：

- `backend/app/tasks/analysis_tasks.py`：原本 mock 期只有 `preprocessing → completed` 两个 stage，现在暴露 `preprocessing / pose / phase / scoring / completed` 五个，通过 `ai_engine` SSE 或轮询上报（**轻量方案**：Celery 任务按固定时间戳更新 stage，不额外加通道 —— docs/14 T4 踩坑里会给判据）
- `backend/app/integrations/ai_engine.py`：识别 `ai_engine` 返回的 `error.code` 映射到 backend 50101-50105 + 退配额
- `client/src/pages/analysis/waiting.tsx`：
  - 去掉"前端时间模拟 stage 推进"那段（当前文件头注释 L7-8 已标记）
  - 仍保留"120s 时间较长提示"（与 mock 一致），不下调到 60s（docs/01 §4.2 最大容忍就是 120s）
- **失败分支真机测**：
  - 上传"黑屏视频" → 50102 / "画质不足"
  - 上传"无人物视频" → 50103 / "未检测到挥杆"
  - 上传"一个人坐着" → 50103

**验收**：
- 正常视频：端到端 < 30s（CPU 基线机），前端 waiting 页 stage 真实推进
- 异常视频：报告页不出现，等待页跳 error 态（复用 M2 现有失败 UI）
- `GET /analyses/{id}/status` 实时反映当前 stage

**依赖/踩坑预警**：
- Celery stage 上报方案：**选"定时戳"而非"真·双向"**，原因是 backend↔ai_engine 加 SSE/Redis pubsub 会拖垮 MVP 复杂度预算；代价是 stage 显示比实际**落后 0.5-2s**，用户无感
- 前端去时间模拟后，如果后端 stage 推进慢，会出现"waiting 页卡在 preprocessing"现象；T4 里加个兜底：**如果 stage 10s 没变，前端自动推进到下一 stage**（纯视觉安抚）
- 配额回退：现有 `analysis_service._refund_on_failure` 已处理，T4 只验证不新增

**影响文件**：
- **改**：`backend/app/config.py`、`backend/app/tasks/analysis_tasks.py`、`backend/app/integrations/ai_engine.py`、`client/src/pages/analysis/waiting.tsx`

```python
def recommend_drills(issues: list[Issue]) -> list[Recommendation]:
    recs = []
    for issue in sorted(issues, key=lambda i: SEVERITY_RANK[i.severity]):
        drills = ISSUE_DRILL_MAP.get(issue.type, [DEFAULT_DRILL])
        # high severity 推 2 条，其它 1 条
        k = 2 if issue.severity == "high" else 1
        for drill_id in drills[:k]:
            if drill_id not in [r.drill_id for r in recs]:  # 去重
                recs.append(Recommendation(
                    drill_id=drill_id,
                    target_issue=issue.type,
                    sort_order=len(recs),
                ))
        if len(recs) >= 3:  # docs/01 §4.3 每次最多 3 条
            break
    return recs
```
