## 一、里程碑定位

MVP 原定 M4（训练计划）/ M5（社交裂变）/ M6（会员支付）是三个独立里程碑，在 W7 一周里**全部收口到最小可交付形态**。取舍原则：

- **不阻塞发布路径**：会员开通 / 打卡 / 邀请绑定是商业闭环必需，优先做
- **不依赖外部审核**：微信支付真实商户号、小程序码生成、朋友圈分享封面，这些都需要小程序审核/ICP/商户号落地，W7 一律**走 mock 或预留**，W8 真机发布前切到真实
- **不扩展重分析**：进步曲线（周度聚合）、家庭版会员、好友动态 —— 全部延后

> W7 的特点：**后端新增 7 张表 + 8 个新 API + 前端新增 2 页 + 替换 7 处占位**。代码量不大，但串起 M4-M6 的**状态机闭环**（订单 → 支付 → 会员激活 → 配额解锁；注册 → 绑定 → 首分析 → 结算邀请奖励）是重点。

## 二、做与不做

### 做（W7 交付）

| 场景 | 必须跑通 |
|---|---|
| 会员开通 | 打开会员中心 → 选月度/年度 → 点「开通」→ mock-pay → 回到首页看到"会员 · 剩余 29 天"标签 + 分析/对话无限 |
| 会员权益 | `analysis_quotas` / `chat_quotas` 查询时识别 `membership_type != 'free'` 且未过期，返回 `remaining=-1` |
| 会员到期降级 | 后台定时作业或惰性检查（`check_membership_on_read`）：到期自动 `membership_type='free'`，配额回到每月 3 次 |
| 训练计划生成 | 分析完成 callback 里同步生成当周 5 个任务（基于 issues × drillLibrary 查表，去重） |
| 打卡 | 训练计划页 → 点任务的「完成」→ 写 `practice_logs` + `training_tasks.status=completed` + `users.current_streak_days` 更新 |
| 邀请绑定 | 新用户注册时若带 invite_code，写 `invitations`（`status='registered'`）+ 给邀请者/被邀请者各 +1 次分析配额 |
| 邀请结算 | 被邀请者完成首次分析后 `status='valid'`；累计 5/10 有效邀请 → 发会员天数（延后也行，T4 先做数据结构 + 5 人奖励） |
| 分享报告 | 报告页 `onShareAppMessage` → 分享卡片到微信聊天（简单 title/path/imageUrl）+ 写 `share_actions` 埋点 |

### 不做（延后 W8 或后续）

| 项 | 原因 |
|---|---|
| 真实微信支付（prepay_id / 签名 / 异步通知） | 需 ICP + 商户号 + 回调公网地址；W7 走 `WECHAT_PAY_MOCK_MODE=true`；代码接口已预留 `wechat_prepay_id / wechat_transaction_id` 字段 |
| 家庭版会员 | DB schema 已支持（`plan_type='family'`），W7 只开 monthly/yearly；UI 不展示 family 卡片 |
| 进步曲线（M4.3） | 需周度/月度聚合 + 图表；数据源（练习日志）W7 已采集，图表留 W8 |
| 小程序码生成（`wxacode.getUnlimited`） | 依赖小程序审核通过 + 服务端接入；W7 前端只展示 6 位邀请码 + 复制按钮，用户通过口令分享 |
| 朋友圈分享 Cover | 需 canvas 离屏绘制 + 图片素材；W7 只做聊天分享（单图 + 标题） |
| 对话历史列表页（M3-T5 延后项） | 与 W7 无直接依赖，归 W8 |
| 续费提醒推送 | 依赖订阅消息模板 + 定时作业；W8 通知体系一起做 |
| 会员挽留弹窗（到期当天） | 同上 |

## 三、技术决策

### 3.1 支付 mock 模式

- 后端：`WECHAT_PAY_MOCK_MODE: bool = True`（默认 True，与 `AI_ENGINE_MOCK_MODE` 同形式）
- 真实模式的占位：后端 `services/payment_service.py` 里 `if settings.WECHAT_PAY_MOCK_MODE: return mock_prepay()` 分支；真实分支 raise NotImplementedError 并带 TODO 注释
- 客户端：`TARO_APP_PAYMENT_MOCK=true` 时 `wx.requestPayment` 不调用，直接弹 modal"模拟支付 3 秒后成功"→ 调 `POST /payments/mock-confirm` → 后端直接跳 `status=paid` → 刷新 userStore
- W8 真实化步骤（留在 docs 里）：填商户号 → 接 WechatPayV3 SDK → 实现 `/payments/callback` 异步通知 + 签名校验 → 客户端 `wx.requestPayment` 联调

### 3.2 会员权益解锁策略

- **不修改 `analysis_quotas` / `chat_quotas` 表结构**（分析/对话两张配额表都已就位）
- 在 `quota_service.py` / `chat_service.py` 的"扣配额"时判断 `user.is_member` helper（membership_type != 'free' AND membership_expires_at > now()）
  - 会员：直接 return（不写配额）
  - 免费：走现有逻辑
- 在"查配额"时若是会员 → `remaining=-1, total=-1`（前端已按此协议处理，M3 时就做了）
- 到期降级：每次读取用户前若发现 `membership_expires_at <= now()` 且 `membership_type != 'free'` → UPDATE 回 'free'（惰性清理，不依赖定时作业）

### 3.3 训练计划生成时机

- **同步生成**（不走 Celery）：`_run_swing_analysis_async` 成功结束后，在同一事务里调 `TrainingPlanService.generate_or_update_weekly(user_id, analysis_id)`
- **增量更新**逻辑：
  - 本周（周一到周日）已有 plan → 在现有 tasks 基础上追加未覆盖的 drill（按 issue 优先级去重）
  - 没有 plan → 新建 plan + 5 个 task（按 issues 排序取 top 3 + drill 库默认 2 条 fallback）
- **task 分配到日期**：今天到周日，按 duration_minutes 均衡（每天 10-20 分钟）
- drill 源：backend 新增 `drills` 表 + 种子数据（与 `client/src/constants/drillLibrary.ts` 对齐，3 条）

### 3.4 邀请码 + 绑定

- `User.invite_code` **已存在**（M1 注册时生成 nanoid 6 位）
- 注册时前端传 `invite_code` 参数到 `POST /auth/login`（onboarding 第一步有 optional 输入框，W7 新增）
- 后端注册逻辑：若 `invite_code` 有效且不是自己 → 写 `invitations` + 奖励两边各 +1 次 `analysis_quotas.bonus`（本月配额 +1，**不影响 `used`**）
- 结算有效邀请：在分析完成 callback 里检查"这是被邀请者的第 1 次分析" → `UPDATE invitations SET status='valid'`；累计 `valid=5` 时（inviter 视角）发 7 天会员（`users.membership_expires_at += 7 days` + `users.membership_type='monthly'`）

### 3.5 分享报告

- Taro 里实现 `usePageScope` 级 `onShareAppMessage` → 返回 `{ title, path, imageUrl }`
- `imageUrl` 复用 `analysis.thumbnail_url`（W6-T3 已落 MinIO；公网可访问，通过 presigned URL）
- `path` 回到报告页带 `?id=ana_xxx&from_share=1`
- 后端 `POST /share/log` 记录 `share_actions`（埋点，不阻塞 UI）
