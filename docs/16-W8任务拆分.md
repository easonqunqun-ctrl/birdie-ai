## 一、里程碑定位

原 W8 定位"正式上线"被拆成两段：

| 里程碑 | 范围 | 外部依赖 |
|---|---|---|
| **W8（本期）** · 核心闭环测试环境就绪 | 合规 + 发布打磨 + 测试部署 + 核心闭环团队内测 | 无（备案/商户号并行，不阻塞 W8） |
| **W9（备案通过后）** · 支付 + 正式上线 | 真实微信支付 + TKE + COS + 审核 + 灰度 | ICP 备案 + 小程序备案 + 商户号 |

**本期决策锚点**：

| 决策项 | 选择 | 影响 |
|---|---|---|
| 备案进度 | **用户自行推进**，不纳入 W8 任务流 | W8 不追踪备案里程碑；W9 启动时再确认 |
| 首版是否含支付 | **不含**。W8 把 UI 入口下架，后端 mock 通路保留 | T4 工作量从"接真实支付"降级为"藏入口 + 走白名单 mock" |
| 部署目标 | **测试环境**（非生产） | TKE / COS / HTTPS 域名全部延后；本期单机云服务器 + Docker Compose 即可 |
| 测试方式 | 团队白名单真机 + 开发者工具 | 不涉及公开审核，不需要合法域名白名单（用预览二维码 / 体验版） |
| 其他端 | App / H5 不发布 | 跨端 adapter 保持现状，RN 占位 TODO 不动 |

**W8 的定位**：**"把已建好的闭环，在生产级依赖（真 AppID / 真 jscode2session / 接近生产的后端）下跑通"**，不是做新功能。代码量约 W7 的 40%，大部分是合规样板 + 发布工程 + 测试环境联调。

## 三、技术决策

### 3.1 测试环境形态

**目标**：**团队白名单真机能访问到一个接近生产的后端**，但不需要备案。

- **部署目标**：1 台腾讯云 CVM（2C4G + 30G 云硬盘足够）+ Docker Compose；或复用本地 Mac + 内网穿透（ngrok / frp）
- **协议**：
  - 方案 A（**推荐**）：CVM 公网 IP + 自签 HTTPS 证书 + 开发者工具勾选"不校验合法域名" → 预览二维码真机可跑
  - 方案 B：ngrok/frp 临时 HTTPS 域名 → 同上；链接不稳定，内测期每次更新需同步
- **数据**：不复用生产数据；测试环境 DB 独立，可以随时 reset
- **限制**：真机体验版**无法绕过合法域名校验**（必须备案通过 + 小程序后台配置才行），因此 W8 内测**必须通过开发者工具"预览"生成的二维码扫一扫进入**，这会打开"开发版"，基础库放松校验

### 3.2 真实微信登录（干掉 mock）

W1 到 W7 一直走 `WECHAT_MOCK_LOGIN=true`。W8 改为：

- 测试环境 `WECHAT_MOCK_LOGIN=false`；填入真实 `WECHAT_APPID` + `WECHAT_SECRET`
- `backend/app/integrations/wechat_auth.py` 实现 `jscode2session(code) → {openid, session_key, unionid?}`（调微信 `api.weixin.qq.com/sns/jscode2session`）
- 保留 mock 分支用于**本地 pytest / 本地 dev**（`APP_ENV=dev` 时自动 mock），测试环境 / 生产一律真实
- 真实 openid 映射到 `users.openid`；老 mock 数据视为测试脏数据，内测前 `make reset` 清一次

### 3.3 支付入口 feature flag 下架

**策略**：代码保留，UI 隐藏。

- 客户端新增 `TARO_APP_PAYMENT_ENABLED`（默认 `false`）；为 `false` 时：
  - `pages/profile/index.tsx` 的"会员中心"入口 → 隐藏或改为"即将开放"（灰态不可点）
  - `pages/index/index.tsx` 配额耗尽 modal → 不再展示"开通会员"按钮，改文案"当前为内测阶段，暂不限量"
  - `pages/coach/index.tsx` L328 / L340 / L502 升级入口 → 隐藏
- `pages/profile/membership` / `pages/profile/invitations`：邀请页保留（邀请是核心闭环的一部分），**会员页留着但入口隐藏**（管理员可以直接 `navigateTo` 进去 mock-pay 给白名单开会员）
- 后端 `/v1/payments/*` 路由**保持可用**（只是前端不走），白名单账号可以用 Postman/curl 调 `mock-confirm` 给自己开会员，方便测试会员分支
- W9 上线前把 flag 翻回 `true` 即可；删除代码不是本期目标

### 3.4 测试环境免费配额放宽

W7 的配额逻辑：免费用户每月 3 次分析、每日 5 次对话；会员无限。

测试环境这量级对内测团队 + 真机走查远远不够。方案：

- 后端新增 `QUOTA_MODE` 环境变量（`strict` / `unlimited`）
- `quota_service.py` 扣配额时 if `settings.QUOTA_MODE == "unlimited"` → 直接放行（不写 `used`，不抛"配额不足"）
- 查配额时 unlimited 模式返回 `{total: -1, remaining: -1}`（沿用会员协议，前端已适配）
- **生产环境必须是 `strict`**，W9 切回
- 测试期白名单 mock-pay 走会员分支也能被覆盖（mock-pay 不受 unlimited 影响，想测会员分支就先 mock-pay 开会员）

### 3.5 隐私合规两层合规

与原 W8 方案不变（本期必做）：

1. 产品侧：《用户服务协议》+《隐私政策》静态页 + 首启拦截同意
2. 微信平台侧：公众平台后台提交《小程序用户隐私保护指引》+ 代码侧挂运行时 API（`getPrivacySetting` / `requirePrivacyAuthorize` / `onNeedPrivacyAuthorization`）

### 3.6 视频内容合规兜底（简化版）

W8 不接腾讯云内容安全（收费 + 接入复杂），**只做小程序官方免费的 `security.mediaCheckAsync`**：

- 客户端 `chooseMedia` 后，把 `thumbTempFilePath` 上传到后端 `POST /v1/security/media-check`
- 后端调微信"小程序安全服务"接口（需 access_token）做图片检测
- 违规：直接返回 fail，客户端 toast "视频不符合规范"，不让提交分析
- 视频本体异步兜底留 TODO（W9 正式上线前视情况补）

### 3.7 最低基础库

- `project.config.json` 明确 `libVersion: "2.27.1"`（微信 2022.12 发布，`chooseMedia` / SSE 都支持）
- `App.onLaunch` 取 `getSystemInfoSync().SDKVersion` 校验；不足即 toast "请升级微信到最新版"

### 3.8 埋点 + 错误上报（最小可用）

- 后端新增 `POST /v1/events` + `events` 表（id / user_id / name / payload jsonb / created_at）
- 客户端 `utils/track.ts`：队列 + 失败重试；核心事件 6 个
- `App.onError` / `onUnhandledRejection` → `track('error_report', {...})`
- 看板：SQL 手查，暂不接 BI
