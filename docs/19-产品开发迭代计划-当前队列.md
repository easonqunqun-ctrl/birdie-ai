### §Q-A1 附 · 叠加 / 按比例退款工单字段模板

> 范围：标准全额退款已在 [backend/app/api/v1/payments.py](../backend/app/api/v1/payments.py) 与 [wechat_pay_v3.py](../backend/app/integrations/wechat_pay_v3.py) 落地；本节为**产品 / 法务**侧仍需逐单评估的场景。U-6 紧急追踪点同条。
>
> 提单去 Lark / Jira 都行，字段沿用本表保持可对账；落库时映射到 `payment_transactions(type='refund')` 与 `orders.membership_end` 回拨。

#### 必填字段

| 字段 | 类型 / 取值 | 说明 / 验收规则 |
|------|-------------|------------------|
| `ticket_id` | str | 工单号；与 Lark/Jira 一致；进入 `payment_transactions.metadata.ticket_id` |
| `user_id` | int | 内部用户 ID（不暴露 openid） |
| `requested_at` | ISO8601 | 用户提单时间；用于 SLA 计算 |
| `orders[]` | list[Order] | 涉及订单列表；按 `paid_at` 升序排列，含「叠加续费」全部订单 |
| `orders[].order_id` | int | 必须存在 `orders` 表；`status ∈ {paid, refunded}` |
| `orders[].paid_amount_fen` | int | 实付金额（分）；与微信账单核对 |
| `orders[].membership_days` | int | 该订单实际贡献的会员天数（来自 `goods.duration_days`） |
| `orders[].consumed_days` | int | **已消耗**的天数（从 `membership_start` 起到工单生效日；按自然日向下取整） |
| `refund_strategy` | enum | `full` 全额退 / `prorated` 按比例退 / `denied` 不退 |
| `legal_basis` | str | 法务依据：用户协议第 X 条 / 平台风控 / 信用卡争议 等；**不可空**，留痕需要 |
| `membership_end_target` | ISO8601 \| null | 退款生效后 `users.membership_end` 应回拨到的目标日；`null` 表示完全降级（与全额退一致）|
| `refund_total_fen` | int | 实际退款总额（分）= Σ `orders[].refund_amount_fen` |
| `orders[].refund_amount_fen` | int | 按 `orders[].refund_strategy` 计算；公式见下 |
| `operator` | str | 客服 / 财务 / 法务负责人；至少一名 |
| `approved_by` | str | 二审签字人（按金额阈值约定）|

#### 按比例退款（`prorated`）金额公式

```
单笔退款额（分） = floor(paid_amount_fen × (membership_days - consumed_days) / membership_days)
```

- `consumed_days >= membership_days` → `refund_amount_fen = 0`（视同已消费完）
- 多笔叠加单：按 `paid_at` 倒序优先退「最晚生效那笔」剩余天数；**不要**跨订单合并按比例，避免与微信 V3 单笔退款的限额冲突
- 用户已有「赠送 / 邀请 / wxacode 兑换」获得的非购买天数：**不计入** `membership_days`，仅消耗顺序参与计算

#### `membership_end` 回拨

| 退款策略 | 回拨规则 |
|----------|----------|
| `full`（全额、单笔订单） | 由 `ensure_membership_valid` + `apply-refund` 标准链路自动降级（已实现）|
| `full`（全额、叠加单中最近一笔） | 回拨到 `membership_end - this_order.membership_days`；不要直接清零 |
| `prorated` | `membership_end_target` 必填，由人工核算后写入；脚本只校验「不晚于原 `membership_end`」 |
| `denied` | 不动 `membership_end`；仅在 `payment_transactions` 记 `refund_status=denied` |

#### 落库映射（仅人工执行时参考；当前**无**自动 API）

```sql
-- 1) 退款记账
INSERT INTO payment_transactions (order_id, type, amount_fen, status, metadata, created_at)
VALUES (:order_id, 'refund', :refund_amount_fen, 'succeeded',
        jsonb_build_object('ticket_id', :ticket_id, 'strategy', :strategy,
                           'legal_basis', :legal_basis, 'operator', :operator),
        now());

-- 2) 订单状态推进（按业务可设为 partially_refunded；MVP 阶段保留 refunded）
UPDATE orders SET status = 'refunded', refunded_at = now() WHERE id = :order_id;

-- 3) 会员到期回拨
UPDATE users SET membership_end = :membership_end_target WHERE id = :user_id;
```

落库前务必**手动**对照 [docs/03-数据库设计文档.md](./03-数据库设计文档.md) §6 的最新字段，部分字段可能已迁移到 `payment_transactions.metadata`。

#### 审计要点（法务）

- 工单审批链与退款金额一一对应；**金额阈值**与二审签字人由内部财务规章约定，本文不固定
- 真实凭据扣减：保留微信商户后台「退款详情」截图至 `docs/release-notes/refund-ticket-evidence/`（**不入库**，按团队规章自管）
- 用户告知：客服话术 + 短信 / 订阅消息（与 `wechat_subscribe_message.py` 模板对齐，待 Q-B1 落地）
