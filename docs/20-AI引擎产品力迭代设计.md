## 二、战略陈述：引擎即护城河

**领翼（小鸟 AI）的长期价值**高度集中在 **`ai_engine` 真推理链路**：从视频到姿态、阶段、特征、评分、诊断与骨骼视频，用户体感上是否「快、准、像教练」，决定留存与口碑。`AI_ENGINE_MOCK_MODE` 仅用于开发/灾备演示；**真实模式**下，产品力来自三条可并行推进的主线（下节），而非「多堆功能」。

**已知典型反馈**（如「世界冠军挥杆得分仅约 30」）多源于 **评分定义与用户直觉尚未对齐** 或 **输入视频与算法假设不一致**，不一定是单点「模型坏了」。本节给出 **可对外统一的话术** 与 **对内迭代机制**，避免舆情与工程各自表述。

## 三、三条主线（产品线）

三条线名称固定为：**可信度（Trust）**、**标尺与公平性（Calibration）**、**教练共识（Coach Consensus）**。迭代时须写明归属哪条线，便于排期与复盘。

### 3.1 可信度（Trust）

**目标**：用户能理解「为什么这样判、在多大程度上可信」；系统在不确定时 **宁可少说、少说错**，并提供 **可行动的回退**（重拍、换机位）。

| 维度 | 用户可见产出（方向） | 工程 / 产品抓手 |
|------|----------------------|----------------|
| **可解释性** | 报告标明关键依据（阶段、角度/轨迹类特征、与哪类典型偏差的对比） | 报告 schema 与文案模板；与 `features.py` / 诊断规则字段对齐 |
| **置信与失败诚实** | 低质量 / 遮挡 / 非完整挥杆 → 明确提示，避免强给高分低分 | 预处理与 `quality_score`、[`docs/01`](01-MVP功能需求规格说明书.md) §4 质量门；错误码见 [`docs/02`](02-API接口设计文档.md) |
| **速度体验** | 在 SLA 内完成；等待页阶段文案与真实阶段大致一致 | W6 性能与队列；[`release-notes/W6-engine-productization-roadmap.md`](release-notes/W6-engine-productization-roadmap.md) |

**验收思路**：抽样视频中，**低置信样本不得**伪装为高置信结论；用户侧「看不懂但觉得武断」的投诉趋势下降。

### 3.2 标尺与公平性（Calibration）

**目标**：综合分与维度分 **在人群内可解释、可纵向对比**；减少「动作好看但分低」「职业球员模板偏差大导致异常低分」的 **无任何解释的割裂感**。

| 维度 | 说明 |
|------|------|
| **量表定义** | 明确综合分/子分项是 **相对「教材理想型模板」的结构契合度**，不等同于赛场成绩、名气或教练主观喜好 |
| **分布与分位** | 在足够样本后，提供 **人群分位** 或 **「与本账号历史相比」** 的主叙事，减轻单次绝对分的争议 |
| **长尾与极端动作** | 对 **个性化极强但仍合规** 的职业动作，通过 **标定集与规则** 降低系统性低估（见第五章） |
| **版本留痕** | 标尺重大变更时 **版本号** 对运营与数据可说明（与 `ai_engine` 发版 / `PHASE_RUBRIC` 变更记录绑定） |

**验收思路**：同一人多次相近挥杆 **分数波动** 在可解释范围内；标定集内教练盲评 **与算法排序** 的方向性一致率提升。

### 3.3 教练共识（Coach Consensus）

**目标**：用 **小规模高质量人类判断** 标定算法与话术边界，专门消化「争议样本」「顶尖样本」。

| 环节 | 说明 |
|------|------|
| **标注协议** | 维度：可打性 / 典型错误标签 / 是否「不宜自动打分」 |
| **争议复盘** | 低分高分与实际观感差异大的样本 **入池复盘**，更新规则或训练权重 |
| **轻量门槛** | MVP 期以 **周级** 例会 + 固定 checklist 为主，不阻塞发版 |

**验收思路**：争议样本池 **关闭率**（已定性、已转工单或已修规则）可追踪。

## 四、评分哲学与对外话术

### 4.1 评分不是什么（避免用户误读）

- **不是**「世界排名」或「这名球员有多强」；**不是**对名人、冠军的社交点评。  
- **不是**球场杆数或比赛成绩的替代指标。  
- **不保证**在 **机位不佳、半身入镜、非全挥杆、强遮挡** 下仍有意义；此时应优先输出 **置信提示** 而非笃定的分数叙事。

### 4.2 评分是什么（推荐对外统一定义）

**综合分及维度分**：在当期算法版本中，表示 **当前视频里可见的挥杆动作，与产品采用的「结构化理想挥杆模型」之间的贴合程度**，兼顾若干可计算的几何与节奏特征；**数值越高表示与模型一致性越好**，便于 **同一人多次拍摄之间的纵向对比** 与 **薄弱维度定位**。  

该定义应在 **首次报告页说明 / FAQ / 客服话术** 中可逐字引用，避免一线口径漂移。

### 4.3 用户沟通模板（可放入产品文案库）

1. **首次出分引导**（示例）：  
   「这个分数表示：在您**本次拍摄且软件能看清的前提下**，您的动作与小鸟 AI 当前采用的标准挥杆结构有多接近。**建议您在相近日、相似拍摄条件下多拍几次**，看趋势比盯单次绝对值更有意义。」

2. **顶尖运动员素材低分**（示例）：  
   「许多职业球员有**强烈的个人技术特点**，在『标准模型』下可能出现**看似偏低的分数**，但不代表其竞技水平。我们正在通过**顶尖样本标定集**持续缩小这种『观感与分数』的割裂。」

3. **重拍 / 机位**（示例）：  
   「本次画面**身体不完整 / 快门角偏低 / 光线过暗**，可能影响关键点检测。**建议侧面全身、光线充足、节拍稳定**后再分析，结果会更稳。」

4. **与教练意见不一致**（示例）：  
   「AI 从**可见几何与节奏**出发；教练可能还会考虑球路、球场策略等。**若结论冲突，以线下教练为准**；您可以把报告中的维度与教练讨论，我们会持续改进模型。」

### 4.4 产品侧落地清单（与 `docs/01` 协同）

| 项 | 说明 | 建议责任 |
|----|------|----------|
| 报告页「分数说明」入口 | 短文案 + 链到 FAQ 或帮助页 | 产品 + 客户端 |
| 低置信 UI | 与质量门 / 错误码一致 | M2 与 §4 质量 |
| 客服 / 运营一页纸 | 与 §4.3 模板一致 | 运营 |

**已列入** [`docs/01-MVP功能需求规格说明书.md`](01-MVP功能需求规格说明书.md) **§4.5**（可编码验收行）；开发计划 **PLAN-ID** 见 [`docs/19-产品开发迭代计划-当前队列.md`](19-产品开发迭代计划-当前队列.md) **§6.3**（主表）与 **§6.5a**（§4.5 已勾存档）：**ENG-01**（分数说明）、**ENG-02**（低置信 / `quality_warnings`）、**ENG-03**（话术一页纸），及 **ENG-04**～**ENG-06**（ECS、纵向叙事、争议复盘）。

## 五、世界顶尖样本标定集（Elite Calibration Set, ECS）

### 5.1 目的

构建 **有授权、有标签、可复现** 的 **顶尖 / 职业代表样本子集**，用于：

- **校准** `scoring.py` / `PHASE_RUBRIC` 与特征权重，减少「职业个性化动作被系统性低估」；  
- **回归测试**：新本版引擎在 ECS 上 **分数漂移** 可量化；  
- **教学叙事**：对内统一「为什么某人某视频是这个分」的培训材料。

### 5.2 数据组成原则

| 类别 | 说明 |
|------|------|
| **来源** | 公开可引用素材需 **版权合规**；合作方 / 选手授权素材需 **书面授权范围** 写明「算法研发与内测」 |
| **多样性** | 男女、不同身形、铁杆/木杆（若画面可辨）、**合规多机位** |
| **元数据** | 尽可能记录：拍摄大致年代、是否减速练习挥杆、画面是否裁切等非标签信息，避免错配期望 |
| **标签** | 至少包含：**教练/评审认为的可打性档**、**是否个性化动作**、**是否不宜自动打分**；可选：与教科书模板的偏差类型 |

### 5.3 工程形态（建议）

| 项 | 说明 |
|----|------|
| **存储** | 与产品一致的对象存储 + **内部清单**（CSV/DB），**默认同盟不外泄** |
| **版本** | ECS **`vYYYYMMDD`**；与 `ai_engine` **Git tag 或大版本** 在发布说明中互链 |
| **门禁** | 核心 MR 或发版前：在 ECS 上跑 **快照对比**（中位数/分位漂移阈值），阈值由产品与算法共担 |
| **与 Mock** | ECS **仅针对真实管线**；Mock 不参与标定 |

### 5.4 产品与法务

- **隐私**：仅限授权素材进入 ECS；用户侧 **无感** 收集不得进入顶尖集。  
- **对外宣传**：「世界冠军」等 **营销用语** 须与真实授权一致，避免合规与期望管理失控。

### 5.5 与现有代码映射（当前仓库）

| 模块 | 路径 | 与 ECS 的关系 |
|------|------|----------------|
| 评分规则 | `ai_engine/app/pipeline/scoring.py` | 权重调整的主落点 |
| 特征 | `ai_engine/app/pipeline/features.py` | 新特征需定义 ECS 上的基线 |
| 真实管线 | `ai_engine/app/pipeline/real_pipeline.py` | 端到端回归入口 |

阶段命名可与发版节奏对齐；以下为 **逻辑顺序**，具体日历见 [`docs/19`](19-产品开发迭代计划-当前队列.md)。

| **E4** | 争议样本周更、教练评审流程工具化（视资源） | Consensus |

## 七、指标与反馈闭环

| 指标 | 说明 |
|------|------|
| **复拍率 / 同周多次分析** | 用户是否愿意信「纵向对比」 |
| **低置信触发率** | 质量门是否过严或过松 |
| **客诉 / 反馈中「分数不公」占比** | 需按是否顶尖样本、是否机位问题 **打标签** |
| **ECS 漂移告警次数** | 发版质量 |

建议在埋点或工单系统中预留 **`feedback_reason`**（如 `score_unfair_pro` / `camera_angle`），便于复盘。
