> **v0.1 已废弃**；**v0.2 战略收敛**；**v0.2.1 增量补强**（职业对比扩展为内容资产 + 新增球友约球轻社交）。完整溯源保留供决策回溯。

| 项 | v0.1 | v0.2 | **v0.2.1（本版）** |
|---|------|------|--------------------|
| **核心命题** | "AI 私教 + 球友生态" | **"把 AI 高尔夫教练做到家"** | 维持 v0.2 |
| **战线宽度** | 9 模块（M7-M15） | 7 模块（M7-M13） | **8 模块（M7-M14）** |
| **第一优先** | M7 球场 / M8 短杆 / M11 画像 / M12 课程 | **M7 引擎 V2 + M8 教练工作台** | 维持 v0.2 |
| **职业球手对比库** | M13 仅"并排播放" | M12 P1（基础对比） | **M12 P1++（扩展为内容资产：球手库 / 专题 / 教学解说 / 差距雷达 / 追平演化）** |
| **球友约球 & 社区** | M14（含 feed / 评论 / 挑战 / 约球） | 整体延到三期 | **拆分**：约球上升到二期 **M13（轻社交 P2）**；feed / 评论 / 挑战赛 / 排行榜仍延到三期 |
| **原 M13 多端扩展** | — | M13 RN 发版 | **M14**（编号顺延） |

**为什么 v0.2.1 这么调？**

1. **职业对比是一击命中心智 + 内容传播抓手**，v0.2 把它当一个并排播放功能太单薄；它应该是"AI 教练的标尺与教学素材库"，与 M7 引擎 V2 协同放大价值。
2. **约球是高频痛点不是社区运营**："一个人练没动力 / 找不到水平差不多的球友"是 C 端真实需求；按"匹配工具"而非"内容社区"做，**克制、轻量、安全可控**，不偏离 AI 教练初心。
3. **明确边界**：约球 ≠ 社区。Feed / 评论 / 点赞 / 挑战赛 / 复杂排行榜 仍在三期，避免内容运营成本拉爆二期。

## 二、二期核心命题：把 AI 高尔夫教练做到家

### 2.1 战略陈述

> **「我们就是一款 AI 高尔夫教练。」**
>
> 不是"AI 工具"、不是"挥杆数据库"、不是"球友社区"、不是"球场服务平台"。
>
> 一切二期投入，都必须能用一句话回答：**「它让 AI 教练更像真教练吗？」** 答不上来的，不在二期范围。
>
> **v0.2.1 补充**：约球 / 职业对比这两类「留存激励」能力，**必须服务于 AI 教练核心闭环**（约球后产生练习 → 触发分析 → 教练 / AI 干预；职业对比作为"标尺"输入用户认知），而**不**作为独立内容生态运营。

### 2.2 一期到二期的真正鸿沟

来自一期内测 / 客服 / 体验版的真实声音，对应一期能力缺口：

| 用户原话 | 暴露的底层问题 | 二期解决方向 |
|---------|---------------|--------------|
| "我用一号木拍的，AI 说我下杆抛杆 65 分，可我打得明明很扎实" | **球杆类型不参与评分**（所有杆共用 PHASE_WEIGHTS / FEATURES） | M7 §4.4.2 球杆差异化标尺 |
| "我换了个角度拍，分数差了 12 分" | **face_on / down_the_line 共用阈值**；机位轻微偏离就漂分 | M7 §4.4.1 机位鲁棒性 / §4.5 阶段 V2 |
| "用 iPhone 240fps 慢动作拍，AI 直接说视频不合规" | **视频读取兼容性窄**（HEVC / 高帧率 / 竖屏 / HDR） | M7 §4.3 视频读取增强 |
| "教练看了我的报告说基本对，但'下杆抛杆'这条不准" | **特征只有 15 个、规则引擎 15 条**，覆盖的 issue 类型有限 | M7 §4.6 特征 V2 + 诊断 V2 |
| "我推杆都是用同一个 app，可你们的 AI 看不懂推杆" | **只支持全挥杆模式**，推杆 / 切杆完全没建模 | M7 §4.7 多场景 + M10 短杆/推杆模块 |
| "教练想给学员留作业，结果还得截图发微信" | **教练 0 触达**，平台没工具 | M8 教练工作台 |
| "我练了 3 周，AI 教练每次都像第一次见我" | **画像太薄**（onboarding 3 题），LLM 上下文不含装备 / 历史 / 偏好 | M9 用户画像 2.0（服务 M7） |
| "练了一堆 drill 不知道自己在哪个水平" | **学习路径缺失**，分析是孤立的 | M11 课程体系 |
| "AI 说我下杆 65 分，可我比 Tiger 差在哪？""能不能让我看一眼林希妤怎么打 wedge？" | **没有职业球手参考**且**没有可浏览的资源库** | M12 职业对比库（**v0.2.1 扩展为内容资产**） |
| "一个人在球馆练俩小时太无聊""有没有水平差不多能一起打的球友" | **0 球友连接** | **v0.2.1 新增** M13 球友约球（**克制版**，非内容社区） |

### 2.3 二期价值主线：四个「更」

> 替代 v0.1 的「更懂用户 + 更解决问题」两条线，更聚焦初心。每个二期模块必须命中至少一条，否则不进。

```
┌───────────────────────────────────────────────────────────────────────┐
│              二期 · 四条价值主线（围绕 AI 教练初心）                    │
├──────────┬──────────┬──────────┬────────────────────────────────────┤
│  更准    │  更全    │  更懂    │  更会教 / 更有伴                     │
│ (Precision) │ (Coverage) │ (Personalization) │ (Teaching & Belonging) │
├──────────┼──────────┼──────────┼────────────────────────────────────┤
│ M7 引擎 V2 │ M7 多场景 │ M9 画像 2.0 │ M8 教练工作台                │
│ · 减少误判 │ · 推杆/切杆│ · 装备/水平 │ M11 课程体系                  │
│ · 视频读取 │ · 多角度  │ · 历史趋势  │ M12 职业球手对比库（**v0.2.1 扩展**）│
│ · 分析能力 │ · 多球杆  │ · 训练偏好  │ M13 球友约球（**v0.2.1 新增**） │
│           │ · 慢动作  │           │                                  │
└──────────┴──────────┴──────────┴────────────────────────────────────┘
```

**优先级排序**：**更准 > 更全 ≈ 更会教 > 更懂**。「更懂」依附于「更准」的真实数据飞轮才有意义。

> v0.2.1 把 "Teaching" 主线扩为 "Teaching & Belonging（更会教 + 更有伴）"，承接 M12 / M13 的留存激励价值；但本质仍服务 AI 教练闭环（见 §2.1 补充）。

## 四、M7 · AI 视频分析引擎 V2（P0 ★★★）

> **本章是二期最重投入的模块**。一期 `ai_engine/app/pipeline/` 已建立 6 阶段 + 15 特征 + 15 条诊断规则的最小可用管线（详 [`docs/20`](20-AI引擎产品力迭代设计.md) §3 / [`docs/05`](05-AI模型技术规格文档.md) §2）；二期沿这条管线做 **3 个方向的纵深升级**：①减少误判 ②提升视频读取能力 ③扩展分析能力。

### 4.1 价值主张

> "让 AI 教练的判断，从『大致没错』升级到『真懂你这一杆』。"

### 4.2 一期引擎能力盘点（基线 · 不要破坏的部分）

| 能力 | 当前实现 | 文件锚点 |
|------|---------|---------|
| 视频归一化 | ffmpeg 转 30fps / 短边 720p / H.264 yuv420p | `ai_engine/app/pipeline/preprocess.py` |
| 姿态估计 | MediaPipe Pose（33 点） | `ai_engine/app/pipeline/pose.py` |
| 阶段分割 | 启发式（主手腕速度阈值 `MIN_MOTION_SPEED=0.008` + Top=y 最小 + Impact=Top 后速度峰） | `ai_engine/app/pipeline/phases.py` |
| 特征提取 | 15 个，覆盖 setup / backswing / top / downswing / impact / follow_through | `ai_engine/app/pipeline/features.py` + `constants.py` |
| 评分 | 理想区间 → 85-100，容忍区间 → 0-84 线性 | `ai_engine/app/pipeline/scoring.py` |
| 诊断 | 15 条 rule，min_confidence=0.6，max_issues=5 | `ai_engine/app/pipeline/diagnose.py` |
| 推荐 | issue → drill 映射，最多 3 条 | `ai_engine/app/pipeline/recommend.py` |
| 质量门 | clarity 80 / stability 0.12 / 帧丢失 10% / 时长 2-30s | `preprocess.py` 阈值 |
| 可观测 | `/health` 暴露 `mock_mode`；CI 跑 ECS v1 stub | `main.py` + `tests/test_ecs_regression.py` |

**二期升级原则**：**所有 V2 能力默认开启**，但通过 `engine_version` 字段灰度切换；V1 至少保留 6 个月作为对照基线。

### 4.3 方向一 · 视频读取能力增强

> "用户拿什么手机拍、什么帧率、什么格式拍，AI 教练都得能看。"

#### 4.3.1 当前限制（从代码确认）

- 仅支持 H.264；HEVC / H.265 视频会被 ffmpeg 降级处理，部分手机录的视频可能直接 fail
- 输入帧率与 30fps 偏差大时（特别是 120 / 240fps 慢动作），归一化后会丢帧 / 重复帧
- iPhone HDR / 10-bit 视频不在测试覆盖范围
- 竖屏视频虽然能跑，但短边 720 上限会把短边压成 720（垂直方向被裁切）

#### 4.3.2 V2 目标矩阵

| 维度 | 当前 | V2 目标 | 落地动作 |
|------|------|--------|----------|
| **编解码格式** | H.264 + yuv420p | + HEVC / H.265 / VP9（兼容性需 ffmpeg `--enable-libx265` + `libvpx`） | Dockerfile 重构 + ffmpeg 全套 codec 编入 |
| **帧率支持** | 30fps 归一化 | 原生支持 24/30/60/120/240fps，统一归一化到 60fps（升级目标 FPS） | `TARGET_FPS = 60` + 高帧率插值降采样 |
| **色彩** | 8-bit yuv420p | + 10-bit HDR / Rec. 2020 → 转 sRGB | ffmpeg `-vf zscale=...` |
| **方向** | 横竖屏均接受，但短边裁切 | 竖屏视频走独立短边阈值（短边 720 → 1080） | preprocess 加方向判定 |
| **分辨率** | 短边 ≤720 | 短边可选 720 / 1080 / 1440（按算法精度需要） | 配置化 |
| **慢动作识别** | 不识别 | 自动识别"录制 240fps 但展示 30fps"的慢动作视频，并归一化到真实时间轴 | 读取 mov 元数据 `nominal_fps` |
| **音频** | 全部丢弃 | 保留（M8 教练语音批注后续可对齐音轨） | ffmpeg `-c:a copy` |
| **质量预检失败原因码** | 50100-50105 共 6 个 | 扩到 20+，覆盖到 codec / format / orientation 维度，每个有具体重拍建议 | `analysisEngineErrors.ts` + `precheck.py` 联动 |

#### 4.3.3 验收口径（二期落地时补 `- [ ]`）

- 推荐机型矩阵（iPhone 12+ / 主流 Android 旗舰 / 千元机抽样 3 款）真机回归 ≥30 个样本，**通过率 ≥95%**
- 不再出现"我视频明明能播放，AI 却看不了"的客诉（按月统计 ≤5 例）

### 4.4 方向二 · 减少误判

> "把『大概觉得你抛杆』改成『带置信度的、能解释清楚的判断』。"

#### 4.4.1 子方向 A · 机位鲁棒性（face_on / down_the_line / 偏角）

| 当前问题 | V2 策略 |
|---------|---------|
| 二个机位共用一套 PHASE_WEIGHTS 和 FEATURES，但实际上：face_on 适合算肩髋旋转 / X-Factor，down_the_line 适合算 swing plane / spine angle | **机位独立标尺**：`PHASE_WEIGHTS_FACE_ON` / `PHASE_WEIGHTS_DTL` 双套，特征 ideal_min/max 各自标定 |
| 用户实际拍摄角度常偏离 30-60° 之间 | **机位识别 V2**：用 setup 帧的肩线水平角度估算机位偏角（±15° 内可视为标准）；偏角过大时给"机位提醒"软警告，不阻断 |
| 一段视频里相机微动（手持） | 加入 **光流补偿**：preprocess 阶段对每帧做全局位移矫正，减少身体关键点的"假位移" |

#### 4.4.2 子方向 B · 球杆类型差异化标尺

| 球杆 | 评分重心应该是 | 当前问题 |
|------|---------------|---------|
| Driver（一号木） | 节奏 / swing plane / 站位较宽 / 击球点上击 | 与铁杆共用 setup 站距阈值，宽站位被扣分 |
| Iron（铁杆） | 杆面方正 / 下击角 / 触球节奏 | 一期主要校准对象，相对准确 |
| Wedge（挖起杆 50-60°） | 半挥幅度 / 重心前置 / 杆面打开 | 当前评分会把短挥误判为"上杆不够" |
| Hybrid / Fairway Wood | 介于 Driver 与铁杆之间 | 同 Driver 问题 |
| Putter（推杆） | 完全不同的特征体系：肩部钟摆 / 头部稳定 / 杆面方正 | **完全不支持**，见 §4.7 + M10 |

**V2 落地**：

- 引入 `club_profiles` 配置：每种球杆有独立的 `PHASE_WEIGHTS_BY_CLUB[club]` + 部分特征的 ideal 范围覆盖
- 报告 UI 上明确"按你的{球杆}应有的标准评分"
- ECS v2 必须按球杆分桶覆盖（每个杆型至少 20 段顶尖样本）

#### 4.4.3 子方向 C · 用户水平差异化标尺（可选 · 风险高）

> 这是 **gradient labeling**，难度大。建议二期 **不做硬切换**，而是用「期望区间分桶 + 文案差异化」温和过渡。

| 用户水平 | 期望话术 |
|---------|---------|
| 初学者（golf_level = beginner） | "对入门球友来说，你的肩部转动已经达到 70°，是非常好的开始" |
| 进阶（intermediate+） | "你的肩部转动 70°，距离职业平均 90° 还有 20° 空间" |

**V2 不动评分数值**，只调用户可见的文案标尺，避免「同一动作不同人不同分」的舆情风险。这条由 LLM 在报告渲染阶段做包装。

#### 4.4.4 子方向 D · 阶段分割算法 V2（从单一启发式升级到多信号融合）

| 当前痛点 | V2 算法 |
|---------|---------|
| `MIN_MOTION_SPEED=0.008` 单阈值不适配慢挥 / 大幅试挥 | **多信号融合**：(1) 主手腕速度峰 (2) 杆头位置（若识别到杆）(3) 双脚压力变化（足部 y 坐标变化代理）(4) 髋部旋转速度 |
| 试挥 / 练习挥被识别为正式挥杆 | **挥杆完整性检测**：必须满足 setup→top→impact→follow 完整序列；不完整 → "未识别到完整挥杆" |
| 短视频（≤2s）经常 `NoSwingError` | 允许 1.5s 起步（之前 `MIN_DURATION_SEC=2.0`），但要求识别到完整阶段 |
| 长视频（含多次挥杆）只识别第一次 | **多挥杆分割**：识别所有挥杆区间，让用户选要分析哪一杆 |

**V2 实现路径**：

1. 保留 V1 启发式作为 fallback
2. 引入 **小尺度时序模型**（轻量 1D CNN 或 Transformer，参考 docs/05 §3.1 Phase 2 计划）训练阶段分类器
3. 训练数据：ECS v2（标注好阶段边界）+ 公开样本数据集
4. 推理阶段：神经网络给软概率 → 启发式做硬约束（如 top 必须在 setup 与 impact 之间）

#### 4.4.5 子方向 E · 置信度上链路化

> 一期 `diagnose` 已经算 confidence 但只用于过滤（<0.6 不展示）；V2 让置信度成为 **UI 一等公民**。

| 改造 | 说明 |
|------|------|
| **每个特征**带置信度 | 关键点 visibility 低 → 特征置信度低 → 评分加灰度标 |
| **每条诊断**带置信度 | 置信度高（>0.85）→ "确诊"；中（0.6-0.85）→ "倾向"；低（<0.6）→ 不展示，但写入"非完美建议"区让用户参考 |
| **整体置信度**写入报告字段 | 进入 swing_analyses 表新字段 `analysis_confidence`，对应 [`docs/20` §3.1 Trust 主线](20-AI引擎产品力迭代设计.md#31-可信度trust) |
| **重拍引导** | 整体置信度 < 0.5 → 报告上方明显提示 "AI 对这次分析的把握不高，建议在更好的拍摄条件下重试" + 一键重拍按钮 |

### 4.5 方向三 · 提升分析能力（新特征 / 新维度）

> "教练眼里看得见的东西，AI 也得看见。"

#### 4.5.1 当前 15 特征的覆盖空白（教研侧反馈）

| 教练常说的 | 当前能算吗 | V2 |
|-----------|-----------|-----|
| 节奏 (Tempo)：上杆与下杆时间比 | ❌ | ✅ 新特征 `tempo_ratio` |
| 重心转移 (Weight Shift)：上杆压右脚 / 下杆压左脚 | ❌（足部 y 间接代理） | ✅ 新特征 `pressure_shift_curve` |
| 动力学接力 (Kinematic Sequencing)：下盘→躯干→手臂→杆 | 部分（downswing_sequence 只看髋肩） | ✅ 4 段速度峰时差 |
| 杆面方正度 (Face Angle at Impact) | ❌ | ⚠️ 二期 best-effort（无球杆识别时退化为"前臂角"代理） |
| 杆头路径 (Club Path) | ❌ | ⚠️ 二期 best-effort（杆识别后才能算） |
| 击球点 (Toe / Center / Heel) | ❌ | ⚠️ 二期不做（需要球高速识别） |
| 杆头速度 (Club Head Speed) | ❌ | ⚠️ 二期不做（需要球杆识别 + 高帧率） |
| 球轨迹 / 落点 | ❌ | ⚠️ 二期不做（需要球追踪 + 户外场景） |

#### 4.5.2 V2 必做的新维度（5 个新特征 / 评分维度）

| 新维度 | 维度名 | 实现 |
|--------|--------|------|
| **节奏 (Tempo)** | `tempo_ratio` = backswing_frames / downswing_frames | 期望 3:1（pro 标准） |
| **节拍稳定** | `tempo_consistency` | 多次挥杆的 tempo 标准差，单次分析下退化为"是否在合理区间" |
| **重心转移** | `pressure_shift_quality` | 用足部 y 坐标 / 髋部 x 偏移代理 |
| **动力学接力** | `kinematic_sequence_quality` | 4 段速度峰时差（pelvis → torso → arms → wrists） |
| **头部稳定性** | `head_stability` | 鼻关键点在整段挥杆中的位移方差 |

**报告 UI 调整**：保留一期六维雷达图（兼容性），增加"动力学详情"折叠区显示新特征，避免一次性吓退用户。

#### 4.5.3 V2 可选的杆 / 球追踪（best-effort）

| 任务 | 实现路径 | 不达标的回退 |
|------|---------|---------------|
| **杆识别** | YOLO 微调（小白盒标注 200 段视频）→ 输出杆头 / 杆身位置 | 失败 → 静默回退，不影响其他特征 |
| **击球瞬间杆面角度** | 杆身角度差分 | 失败 → 显示"未能可靠识别杆面" |
| **swing plane** | 杆头 3D 轨迹平面拟合 | 失败 → 不展示 |

二期定位：**做到能展示给用户惊艳的"看到了杆"**，但 **不**承诺所有视频都能识别。报告 UI 加 "此功能仍在打磨" 标签。

### 4.6 方向四 · 诊断规则 V2

| 当前 15 条规则的问题 | V2 方向 |
|--------------------|--------|
| 规则与特征强耦合，新特征加进来要改 if-else | 抽象 `RuleEngine`，每条规则 = 一个 `Rule(name, conditions, severity, confidence)` 配置 |
| 互斥逻辑通过阈值隐式保证，难维护 | 显式声明 `mutually_exclusive_with` 字段 |
| 文案与规则在同一文件 | 文案独立到 i18n 字典；规则只产出 `issue_type + payload` |
| 严重度静态 | 严重度按"偏离理想区间的相对距离"线性映射，更细腻 |
| 规则数 15 | 二期目标 25-30（覆盖更多 issue 类型，如 "lift_off_back_foot" / "casting_late" 等细分） |

### 4.7 方向五 · 多场景分析（推杆 / 切杆 / 全挥）

> 与 **M10 短杆 & 推杆 AI 分析** 是同一个能力的不同表层；M7 是引擎能力，M10 是产品打包。

| 场景 | 一期 | V2 |
|------|------|-----|
| 全挥杆 | ✅ | 维持，并接 §4.4 球杆差异化 |
| 推杆 | ❌ | 引擎层新增 `mode=putting`：特征改为肩部钟摆角 / 头部稳定 / 推杆面方正 / 推杆节奏 |
| 切杆 | ❌ | 引擎层新增 `mode=chipping`：半挥幅度 / 杆面打开 / 击球点 |
| 沙坑救球 | ❌ | 二期 **best-effort**，先按切杆模式跑 |
| 试挥（练习挥） | 误判为正式挥 | 识别为 "practice swing"，不进入正式分析记录 |

引擎层暴露 `mode` 参数；客户端在 `pages/analysis/params` 让用户选「分析模式」（与球杆联动：选 Putter 自动切 putting mode）。

## 五、M8 · 教练工作台（P0 ★★★）

### 5.1 价值主张

> "让一位高尔夫教练用过一次后，下次上课主动拿出来用。"
>
> 教练满意 = 我们的最佳分发节点（一个教练带 10-30 学员）+ 最佳内容贡献者（验证 AI 输出 / 提供专业话术）+ 最强信任背书。

### 5.2 用户场景

| 场景 | 流程 |
|------|------|
| **教练入驻** | 教练扫码 → 教练身份切换（与现 C 端账号同 user_id，加 `coach_profile`） → 上传教学资质 → 平台审核（24h 内） → 通过后开启工作台 |
| **课中辅助（黄金场景）** | 教练在球馆给学员上课 → 用小程序拍学员挥杆 → 当场看 AI 报告 → 用 **语音 30 秒 + 涂鸦标注** 批注（"上杆这里再慢一拍"） → 一键发到学员小程序 |
| **课后作业** | 教练根据本节课问题 → 从 drill 库挑 2-3 个动作（或上传自己录制的私教视频） → 设定打卡天数 → 学员小程序收到通知 + 推送 |
| **学员看板** | 教练打开学员列表 → 看每个学员最近 7 天活跃度 / 最近一次分析评分 / 作业完成度 → 对长期不练的学员一键发提醒 |
| **教学报告自动生成** | 一节课多个学员的挥杆分析汇总 → AI 给教练一份"本课学员问题汇总 + 下节课重点建议"，可导出 PDF 发学员 |

### 5.3 功能矩阵

| 子模块 | 说明 | 与一期复用 | 二期新增 |
|--------|------|------------|---------|
| **教练账号体系** | 同 user_id + 加 `coach_profile`（资质照片 / 教练等级 / 简介 / 专长）；教练身份是 user 的"扩展属性"，不是独立账号 | `users` 表 | `coach_profiles` + `coach_verifications` |

## 八、M11 · 课程体系 & 学习路径（P1）

### 8.1 价值主张

> "把零散的 drill 升级成 7 阶段学习路径，让用户知道『我现在在哪一级、下一步学什么』。"

### 8.2 与 v0.1 的差异

v0.1 的 M12 课程体系，**v0.2 维持其内容设计**，但调整定位：

- v0.1：作为"更解决问题"主线的独立 P0 模块

### 8.3 7 阶段课程（与 v0.1 一致 · 借鉴 PGA 培训体系）

| 阶段 | 重点 | 通关考核（要求 M7 分析支持） |
|------|------|-----------------------------|
| 1 · 入门 | 握杆 / 站位 / 瞄准 | setup 阶段评分 ≥ 80 |
| 2 · 全挥杆基础 | 上杆 / 顶点 / 平衡 | 综合评分 ≥ 60 |
| 3 · 下杆与击球 | 下杆顺序 / 杆面方正 / 击球点 | downswing 评分 ≥ 70 |
| 4 · 距离控制 | 杆型选择 / 半挥 / 节奏 | 个人 yardage 误差 ≤ 5 码（M10 + M9） |
| 5 · 球路控制 | draw / fade / 弹道高度 | （二期 best-effort，依赖 M7 §4.5.3 杆识别） |
| 6 · 短杆 & 推杆 | 切杆 / 推杆 / 沙坑 | 短杆 / 推杆模式（M10）评分达标 |
| 7 · 下场实战 | 球场策略 / 心态 / 体能 | **三期再开**（依赖下场场景） |

> 二期开放 1-6 阶段，第 7 阶段在三期接入下场后开放。

### 8.4 功能矩阵

| 子模块 | 说明 |
|--------|------|
| **课程库** | 每阶段下 5-10 节课，每节课含目标 / 视频 / drill / 自测 |
| **学习路径** | 用户登录后看到"你在第 3 阶，还差 2 节课通关" |
| **阶段考核** | 关卡式：要求拍一段视频 → M7 引擎给评分 → 达标即升阶 |
| **证书 / 勋章** | 通关有徽章（对齐白皮书 §7.2 金色系） |
| **教练定制课程**（与 M8 联动） | 教练可在工作台上传自己的微课，关联学员训练计划 |

### 8.5 与一期 / M7 的复用

- **复用**：drill 库（13 → 25-30）+ M2 / M7 分析（用于阶段考核）+ M3 LLM（用于课程互动）
- **不破坏一期**：训练计划仍存在；课程体系是"学习路径"层，drill 仍是底层资产

## 十、M13 · 球友约球 & 轻社交（P2 · v0.2.1 新增）

### 10.6 隐私 / 安全 / 合规（v0.2.1 重点）

> 约球是平台**第一次让用户之间直接接触**，必须从一开始就把红线划清楚。

| 风险 | 控制措施 |
|------|---------|
| **位置信息泄露** | M9 §6.5 已说明位置敏感信息需要独立授权；M13 加二次提醒："你的常去球馆将向匹配球友可见" |
| **隐私 - 真实身份** | 匹配阶段只展示脱敏信息（差点段 / 距离段 / 头像）；只有双方接受后才生成一次性微信群链接（不存对方 openid / 手机号） |
| **不当邀请 / 骚扰** | 每个用户每日发起邀请上限 5 个；连续被拒 3 次进入 24h 冷却；接受率长期低于阈值的账号触发风控审核 |
| **No-show / 失约** | meetup_feedbacks 互评 + 信用积分 + 信用积分低于阈值不可发起邀请 |
| **未成年人保护** | 14 岁以下账号不开放 M13 入口（依赖一期账号实名 / 年龄字段，若一期未做需先补） |
| **女性球友安全** | 默认不展示性别给陌生人；女性可主动设置"仅匹配同性"或"仅匹配教练" |
| **场地真实性** | venues 表 ugc 数据走运营审核；地址 / 名称错误时用户可一键举报 |
| **平台不撮合交易** | 严格表述"平台只做信息匹配，不参与任何线下活动 / 不承担任何责任"；服务协议同步 |
| **内容安全（自我简介 / 互评 / 自助赛简介）** | 走 docs/06 + 内容安全 API；命中关键词 / 黄反图自动隐藏 |
