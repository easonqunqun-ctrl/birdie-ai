# real_pipeline 生产 Runbook（Q-B4）

> 目的：CVM 上 `ai_engine` 已默认 **real**（`AI_ENGINE_MOCK_MODE=false`），本文件给值班人员一份**可直接对照排障的操作手册**：异常如何识别、按什么顺序处置、何时回退到 mock。  
> 关联：[`W6-engine-productization-roadmap.md`](./W6-engine-productization-roadmap.md) · [`W6-real-engine-walkthrough.md`](./W6-real-engine-walkthrough.md) · [`docs/19` §6.3 Q-B4](../19-产品开发迭代计划-当前队列.md#63-主表plan-id)

## 0. TL;DR（30 秒）

| 现象 | 第一动作 |
|------|----------|
| 用户报告「上传后一直转圈」 | 看 `swing_analyses` 最近 10 分钟 `status='processing'` 数量；> 5 即 P1 |
| `/v1/health` `ai_engine.reachable=false` | 重启 `xiaoniao-ai-engine` 容器，1 分钟内未恢复则**回退 mock**（§5） |
| 50100 在最近 10 分钟占失败 > 30% | backend → ai_engine 网络问题；查 docker network / 端口 |
| 50101～50105 突增同一码 | 引擎自身或素材异常；按 §3 表分类，**不要**盲目重启 |
| 任一异常导致 P0 | 立即按 §5 切 mock，恢复用户主链路，再排根因 |

## 2. 监控触发条件（值班关注）

| 信号 | 阈值 | 数据源 |
|------|------|--------|
| `swing_analyses.status='processing'` 持续 > 5 分钟 | 单条即可告警 | DB `WHERE status='processing' AND updated_at < NOW() - INTERVAL '5 minutes'` |
| 10 分钟内 `failed` 占 `started` > 30% | 区分 50100 与 50101～50105 | DB `swing_analyses.error_code` |
| `/v1/health` `ai_engine.reachable=false` 连续 2 次 | 立即 P1 | curl 或外部探测 |
| `xiaoniao-ai-engine` 容器 RSS > 1.5GB 或 OOM Kill | 立即重启 | `docker stats` |

未接入 Sentry / Prometheus 之前，运维至少每天 10:00 / 22:00 各扫一次 health + 最近 24h 错误码分布。

## 3. 错误码 → 处置矩阵

| 码 | 含义 | 频次 | 处置 |
|----|------|------|------|
| **50100** | backend ↔ ai_engine transport 不可达（3 次重试耗尽） | 罕见 | ① `docker ps` 看 `xiaoniao-ai-engine` 是否在；② `docker logs xiaoniao-ai-engine --tail=200`；③ `docker network inspect xiaoniao-network` 检查 backend / ai_engine 同网络；④ 1 分钟内未恢复 → 回退 mock（§5） |
| **50101** | ffmpeg 解码 / 视频下载失败 | 中 | 多为用户视频损坏 / MinIO 拉不到；看 `xiaoniao-ai-engine` 日志中 `PreprocessError` 上下文；若指向 MinIO，复查 `xiaoniao-minio` 健康 |
| **50102** | 画质不足（拉普拉斯方差 < 50 或亮度过低） | 中 | **用户问题**，不需处置；批量同模式手机突增需找产品复盘文案 |
| **50103** | 有效人体帧 < 70% | 中 | 同 50102；产品维度——若某机型 / 场景集中触发，归到「拍摄指南」迭代 |
| **50104** | 检测到人但无有效挥杆 | 低 | 用户拍错；无需处置 |
| **50105** | MediaPipe 模型加载 / 推理异常 | 罕见 | **要关注**：可能是镜像 / 权重坏；查 `docker logs xiaoniao-ai-engine` 启动段；如反复出现 → §5 回退 mock |

50101～50105 **全部退配额**，用户侧不会无端扣额度。运营不必为单次失败安抚用户。

## 4. 常规排障步骤

### 4.1 快速 triage（按顺序，不要跳）

```bash
# 1. 外网 health
curl -sfS https://api.birdieai.cn/v1/health | jq

# 2. 容器健康
ssh ubuntu@1.13.198.172 'docker ps --format "{{.Names}}\t{{.Status}}" | grep xiaoniao'

# 3. ai_engine 最近 200 行日志
ssh ubuntu@1.13.198.172 'docker logs xiaoniao-ai-engine --tail=200'

# 4. 最近 1 小时 backend 调用 ai_engine 异常
ssh ubuntu@1.13.198.172 'docker logs xiaoniao-backend --since=1h 2>&1 | grep -E "ai_engine_retry|50100|50105"'
```

### 4.2 单条 analysis 复盘

```bash
ssh ubuntu@1.13.198.172
docker exec -it xiaoniao-postgres psql -U postgres -d xiaoniao -c \
  "SELECT id, status, stage, stage_progress, error_code, error_message, quota_refunded,
          created_at, updated_at
   FROM swing_analyses WHERE id='ana_xxx';"
```

字段含义：
- `stage` / `stage_progress`：值班只读字段；终态失败时 backend 会清成 `null/0`
- `quota_refunded`：50100～50105 应为 `true`；若为 `false` 是 bug，记 issue
- `error_message`：ai_engine 透传，可贴给用户做安抚

### 4.3 重启 ai_engine（不切 mock）

```bash
ssh ubuntu@1.13.198.172
cd ~/lingniao-golf
docker compose -f docker-compose.yml -f docker-compose.cvm.yml \
  --env-file .env.local restart ai_engine
# 等 30s，再 curl health 确认 mock_mode=false 且 reachable=true
```

**注意**：仅重启 `ai_engine`，不要重启 backend / worker / beat，避免误伤进行中的分析。

## 5. 回退到 mock（应急 ONLY）

**触发条件**（满足任一）：
- `xiaoniao-ai-engine` 重启 1 分钟内 health 仍 fail
- 50105 在 10 分钟内出现 ≥ 3 次（模型层坏）
- 整体失败率（任意码）> 50% 且持续 5 分钟以上
- 即将进入大促 / 媒体投放窗口，引擎不稳

**操作**：

```bash
ssh ubuntu@1.13.198.172
cd ~/lingniao-golf
# 1. 编辑 .env.local：AI_ENGINE_MOCK_MODE=true
sudo sed -i 's/^AI_ENGINE_MOCK_MODE=false/AI_ENGINE_MOCK_MODE=true/' .env.local
grep AI_ENGINE_MOCK_MODE .env.local   # 自检为 true

# 2. 只重启 ai_engine 即可（backend 不消费此变量）
docker compose -f docker-compose.yml -f docker-compose.cvm.yml \
  --env-file .env.local up -d ai_engine

# 3. 验收
curl -sfS https://api.birdieai.cn/v1/health | jq '.ai_engine'
# 期待：{"reachable": true, "mock_mode": true}
```

**副作用**：
- 所有新建分析报告将由 mock pipeline 生成，**分数和问题为固定 fixture**，对用户呈现「正常报告体验」但**不代表真实挥杆**
- 历史已完成的真分析报告不受影响
- 配额、计费、提醒等周边逻辑全部不变

**回切到 real**：
- 与 §5 反操作；最好选用户低峰（凌晨 03:00～05:00）
- 切完 30 分钟内人工抽测 1～2 条真实挥杆视频，确认 6 维 / drill / 骨骼视频均正常

**复盘要求**：每次回退必须在 [`docs/09`](../09-运营支撑文档.md) 留 5 行纪要：触发时间 / 触发条件 / 处置耗时 / 根因 / 是否进 backlog。

## 6. 升级 / 发版注意

- ai_engine 改了 `pipeline/*.py` 或 `requirements/uv.lock` → 必须本地 `make ai-engine-smoke` 通过再 push
- 改完 push → `make release-cvm` 会自动 `compose up -d --build`，无需单独操作
- 发版后 5 分钟内：
  1. `curl /v1/health` 确认 `ai_engine.reachable=true`
  2. 真实视频抽测 1 条（运营人员手机即可），看完整 6 维与骨骼视频
  3. 若任一项失败 → §5 回退

**禁止**直接在 CVM 上手改 `ai_engine` 源码或在容器内 `pip install`；任何热修都必须走 git → push → release。

## 7. 已知限制（要告诉客服）

- **首条真分析延迟**：MediaPipe 冷启动需要加载模型，单 worker 首次 8～12s，比 mock 慢
- **复杂场景失败率**：低光 / 远景 / 多人 / 不完整挥杆都会进 50101～50105；这是产品**有意**的退场，**不是 bug**
- **暂不支持**：杆头检测、教练校准评分、断点续传（见 [`docs/01` §4.2](../01-MVP功能需求规格说明书.md)）
